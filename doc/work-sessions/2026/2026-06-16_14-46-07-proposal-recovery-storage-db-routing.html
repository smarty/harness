Route Recovery's SQL through storage.DB

All cursor state and paging logic stay in Recovery; only the two SQL queries move to storage + mysql.Mapper. No *sql.DB / *sql.Rows crosses the storage contract.
Proposal · 2026-06-16 14:46:07 · branch master · author Michael Whatcott
Status: DRAFT — awaiting review. No code to be written until approved.

Background

Dispatcher and Writer now route their database work through the storage.DB seam (Handle(ctx, operation any) error), implemented by mysql.Mapper. Recovery is the last sqladapter type still holding a *sql.DB and composing raw SQL. This proposal moves its two queries behind the same seam.

Recovery is different from the other two: it is a stateful keyset cursor, not a thin shim. Per your direction, all of that state and logic stays put — only the SQL leaves. Today (recovery.go) it owns:

  • cursor, boundary, snapshotted — the paging state.
  • snapshot-once: on the first Recover it freezes the backlog's id bounds (MIN/MAX of undispatched rows that predate startup).
  • completion: cursor >= boundary ⇒ return the empty page (backlog exhausted).
  • advance-only-after-a-clean-scan: the cursor moves to the last returned id only when a page comes back non-empty and error-free, so a failed page is re-served, never skipped.

Exactly two things are SQL, and only these move:

snapshot: SELECT MIN(id), MAX(id) FROM Messages WHERE dispatched IS NULL page: SELECT id, type, payload FROM Messages WHERE dispatched IS NULL AND id > ? AND id <= ? ORDER BY id LIMIT ?

Approach

1 — Two result-bearing storage operations

These are the first operations that carry output back to the caller. The mechanism is the one already in use (the Mapper mutates fields of the operation the caller passes by pointer — as InsertMessages does with assigned IDs); here the mutated fields are query results. Neither type imports database/sql, so the storage package stays driver-agnostic. 

// internal/storage/operations.go

// LoadUndispatchedBounds reports the id range of the undispatched backlog.
type LoadUndispatchedBounds struct {
    Found bool   // false when the backlog is empty (both bounds NULL)
    Min   uint64 // populated by the handler
    Max   uint64
}

// LoadUndispatchedPage loads up to Limit undispatched messages in id order
// within the half-open window (AfterID, ThroughID].
type LoadUndispatchedPage struct {
    AfterID   uint64                 // exclusive lower bound (the cursor)
    ThroughID uint64                 // inclusive upper bound (the frozen boundary)
    Limit     int
    Messages  []*contracts.Message // populated by the handler
}

2 — The Mapper handles the queries

Two new cases in Handle. All driver detail — QueryRowContext/QueryContext, sql.NullInt64, *sql.Rows, the bytes.Buffer payloads, and the hard-coded application/json content type — lives here, in the package that already owns database/sql. 

case *storage.LoadUndispatchedBounds:
    return this.loadUndispatchedBounds(ctx, op)
case *storage.LoadUndispatchedPage:
    return this.loadUndispatchedPage(ctx, op)
func (this *Mapper) loadUndispatchedBounds(ctx context.Context, op *storage.LoadUndispatchedBounds) error {
    var lo, hi sql.NullInt64
    row := this.handle.QueryRowContext(ctx, `SELECT MIN(id), MAX(id) FROM Messages WHERE dispatched IS NULL`)
    if err := row.Scan(&lo, &hi); err != nil { return err }
    if lo.Valid && hi.Valid {
        op.Found, op.Min, op.Max = true, uint64(lo.Int64), uint64(hi.Int64)
    }
    return nil
}

func (this *Mapper) loadUndispatchedPage(ctx context.Context, op *storage.LoadUndispatchedPage) error {
    rows, err := this.handle.QueryContext(ctx, `SELECT id, type, payload FROM Messages
        WHERE dispatched IS NULL AND id > ? AND id <= ? ORDER BY id LIMIT ?`, op.AfterID, op.ThroughID, op.Limit)
    if err != nil { return err }
    defer func() { _ = rows.Close() }()
    for rows.Next() {
        var ( id uint64; typeName string; payload []byte )
        if err := rows.Scan(&id, &typeName, &payload); err != nil { return err }
        op.Messages = append(op.Messages, &contracts.Message{
            ID: id, Type: typeName, Content: bytes.NewBuffer(payload), ContentType: "application/json",
        })
    }
    return rows.Err()  // a mid-scan error returns non-nil; the caller will not advance
}
The clean-scan guarantee survives the move The Mapper either fully populates op.Messages and returns nil, or it returns an error (from QueryContext, Scan, or rows.Err()). Recovery advances its cursor only on nil-with-non-empty-page, so a failed scan still leaves the cursor untouched and the page is re-served on retry — identical to today. Each op is freshly allocated per call, so a partially-populated op.Messages on an errored page is simply discarded.

3 — Recovery keeps all state & logic; only the SQL calls become operations

Recovery swaps *sql.DB for storage.DB. The cursor/boundary/ snapshotted fields and every branch of the paging logic remain exactly where they are. The two private methods stop issuing SQL and instead build an operation, hand it to db.Handle, and read the results back off the operation. 

type Recovery struct {
    db          storage.DB
    cursor      uint64
    boundary    uint64
    snapshotted bool
}

func NewRecovery(db storage.DB) *Recovery { return &Recovery{db: db} }

// Recover is UNCHANGED — same snapshot-once / boundary / page sequencing:
func (this *Recovery) Recover(ctx context.Context, limit int) ([]*contracts.Message, error) {
    if !this.snapshotted {
        if err := this.snapshot(ctx); err != nil { return nil, err }
    }
    if this.cursor >= this.boundary { return nil, nil }
    return this.page(ctx, limit)
}

snapshot and page keep the cursor arithmetic (cursor = Min - 1, advance to last id) and just read it from the operation instead of from sql.Row/sql.Rows. No buffer reuse: recovery is a one-time startup procedure, so each call allocates a fresh operation (the existing code comment already declines pooling here).

Data flow

Call sqladapter — Recovery (state + logic) storage — operation (pure data) mysql — Mapper (owns *sql.DB / *sql.Rows)
snapshot db.Handle(op), then if op.Found: cursor = op.Min-1, boundary = op.Max LoadUndispatchedBounds{} → handler fills Found, Min, Max SELECT MIN(id), MAX(id) … WHERE dispatched IS NULL
page db.Handle(op), then if non-empty & no error: cursor = last msg id LoadUndispatchedPage{AfterID, ThroughID, Limit} → handler fills Messages SELECT id, type, payload … id > ? AND id <= ? ORDER BY id LIMIT ?

Boundary by column: cursor/boundary state never leaves sqladapter; storage carries only data (no database/sql); *sql.DB/*sql.Rows never leave mysql.
