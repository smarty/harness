A generic Decorator seam for request-context-driven message enrichment

2026-06-22 17:04:41  ·  github.com/smarty/harness/v2  ·  branch seam-context-meta

This is 1 of 3 coordinated proposals. The other two live in the message-registry/v4 and billing-context (root) repos. This document covers the open-source, general-purpose piece: a new extension point in the harness pipeline. It deliberately contains no knowledge of Meta, message-registry, or HTTP headers — that coupling stays in the consuming application.

Background

The harness drives a store-and-forward pipeline: entrypoint → execution → serialization → persistence → completion → broadcast → terminal. Domain handlers (Execute<T>(cmd, broadcast func(...any))) are deliberately pure — they never see a context.Context, and we want to keep it that way.

The consuming application needs to attach request-scoped data to every message a command produces before that message is serialized and persisted (the persisted bytes are what recovery re-dispatches, so doing it later — e.g. in the Dispatcher — is too late and would leave the stored copy incomplete). Today there is no hook between "domain produced a value" and "value gets serialized," and the request context.Context that arrives at the entrypoint is discarded:

internal/pipeline/00_entrypoint.go:67 
func (this *entrypoint) Handle(_ context.Context, messages ...any) {  // ctx ignored!
internal/pipeline/contracts.go:69 
type batch struct {
    instructions []any
    complete     func(stored bool)
}  // no ctx field — the request context never travels with the work

We need the harness to (a) carry each command's context forward and (b) expose a generic seam where a caller-supplied collaborator can transform the produced values — without the harness learning anything domain-specific.

Approach

Add a fourth optional collaborator — Decorator — alongside the existing Serializer, Dispatcher, and Monitor. It follows the same pattern: an interface in contracts, an Options.Decorator(...) setter, and a no-op default so existing callers are unaffected.

contracts/contracts.go (new) 
// Decorator transforms the message values a domain handler produced, before
// they are serialized. It receives the context that accompanied the originating
// command at the entrypoint, so request-scoped data can be applied per message.
//
// Implementations MUST return a slice of the same length, each element either
// the original value or a replacement of the SAME concrete Go type: the
// pipeline has already derived each message's registered Type name from the
// pre-decoration value. The default no-op Decorator returns messages unchanged.
type Decorator interface {
    Decorate(ctx context.Context, messages []any) []any
}

Where the seam fires — and why granularity matters

The single most important design detail: a batch (one Handle(ctx, …) call) carries one context, but the execution stage coalesces multiple batches into a single unitOfWork (up to ExecutionUnitSize, default 64). So the context must ride on the batch, and decoration must run per-batch over exactly the slice of results that batch produced — never at unit granularity, or two requests' contexts would cross-contaminate.

Why this satisfies all three constraints

  • harness stays general: it depends only on context.Context + an interface it defines. No Meta, no registry.
  • domain stays context-free: the Execute<T> signature is untouched; decoration runs in a stage the harness owns.
  • before serialization & persistence: the seam fires inside the execution stage, upstream of both.

for batch := range this.input {
    // ... unit allocation unchanged ...
    unit.completions = append(unit.completions, batch.complete)
    start := len(unit.results)
    for _, instruction := range batch.instructions {
        this.executor.Execute(instruction, func(outgoing ...any) { /* append to unit.results */ })
    }
    this.applyDecorator(batch.ctx, unit.results[start:])   // ← new
    // ... flush logic unchanged ...

func (this *execution) applyDecorator(ctx context.Context, produced []*contracts.Message) {
    if len(produced) == 0 { return }
    this.values = this.values[:0]
    for _, m := range produced { this.values = append(this.values, m.Value) }
    this.values = this.decorator.Decorate(ctx, this.values)
    for i := range produced { produced[i].Value = this.values[i] }
}

Trade-offs & open questions

TopicAssessment
Backward compatibilityFully compatible. Decorator defaults to the shared nop; zero-option pipelines behave exactly as before.
Message Type integrityMessage.Type is derived from reflect.TypeOf(result) at creation — before decoration. The interface contract requires the replacement value keep the same concrete type. (Our Meta use respects this: SetMeta returns a copy of the same struct type.)
Misbehaving decorator (wrong length)Open question. Options: (a) document the same-length contract and index defensively; (b) skip write-back if len differs; (c) Monitor.Track a new observation + panic, mirroring serialization's fail-fast. Recommendation: defensive guard that no-ops on mismatch, no new observation type for v1.
Per-batch overheadOne extra slice reset + copy per batch when a decorator is set; a single bounds check when it is the nop. Negligible vs. serialization/IO. The reusable this.values buffer avoids per-batch allocation.
Pool hygienebatch is pooled; holding a request ctx after completion would pin its values. Clear item.ctx = nil in the complete closure (extend pipeline_pool_hygiene_test.go).
Consumer/replay pathsThe BlockingEntrypoint (MQ consume, cron) passes a context without request data. That is correct and expected — the decorator simply finds nothing request-scoped to apply. Documented on the interface.

Your Turn

The hands-on "Your Turn" for this effort is the pure enrichment function in the message-registry/v4 repo — meta.Decorate(ctx, now, messages) — documented in that repo's companion proposal. It is the small, self-contained piece of real logic (read context, stamp timestamp, coalesce metadata onto each event). The harness changes in this document are mechanical plumbing and will be agent-implemented so the seam is ready for your function to plug into.
