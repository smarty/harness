Proposal: Paged Recovery — resolution for F7, github.com/smarty/harness/v2

2026-06-12 · responds to F7 of the adversarial pre-release review · status: DRAFT — awaiting approval; no code until approved

Background

F7 (MEDIUM): the recovery path loads the entire undispatched backlog into memory in one shot. Two compounding causes:

  • sqladapter/recovery.go:22-58 — the SELECT … WHERE dispatched IS NULL ORDER BY id has no LIMIT, and the row loop appends every row (full payload) into one slice before returning.
  • internal/pipeline/recovery.go:38-48 — Recovery.Listen chunks the output of that one call into recoveryBatchSize=64 units; the chunking bounds unit size, not resident memory.

Startup memory is therefore O(backlog). With mediumblob payloads up to 16 MB, a few thousand rows stranded by an extended dispatcher outage can be gigabytes — and the failure mode is self-reinforcing: the bigger the backlog, the more likely recovery OOMs at startup, which prevents the dispatching that would shrink the backlog. A crash-loop with no exit.

Approach: pull-based paging — Recover(ctx, limit) called until empty

Invert recovery from "one giant pull, then chunk" to "pull one unit-sized page at a time, until empty." The pipeline's existing channel backpressure then bounds resident memory at a handful of pages no matter how large the backlog is.

1. Contract change (contracts/contracts.go)

// before
Recoverer interface {
    Recover(context.Context) ([]*Message, error)
}

// after
Recoverer interface {
    Recover(ctx context.Context, limit int) ([]*Message, error)
}

New contract, to be spelled out in the interface doc comment:

  • The pipeline calls Recover repeatedly; each successful call must return the next page (at most limit messages, in dispatch order) of the backlog as it existed at startup — never messages already returned by a prior successful call.
  • An empty result means recovery is complete; the pipeline stops calling and opens the gate to live traffic.
  • An error must not lose ground: the pipeline retries (backoff, as today), and the implementation must re-serve the failed page rather than skip it.
  • Implementations are stateful cursors, invoked from a single goroutine (consistent with Writer/Dispatcher — see F10).
Why change the signature instead of just documenting loop-until-empty on the old one? The old signature with silently-changed semantics is a trap: any existing implementation that returns the full backlog on every call would re-publish in-flight rows on each iteration (duplicates bounded only by dispatcher progress, unbounded if dispatch is down). Adding the limit parameter makes every implementation fail to compile, forcing each author to confront the paging contract once. The module is pre-tag, so this is not a breaking release. A side benefit: the page size stays a pipeline-owned constant (recoveryBatchSize=64) passed in per call — the adapter needs no configuration of its own, and one page is exactly one unit of work.

2. Pipeline loop (internal/pipeline/recovery.go)

Listen becomes a page loop; the retry/backoff logic is unchanged in spirit but moves inside it:

  • Loop: recover one page (retrying with backoff on error, exactly as today); forward it as a unit on output; repeat until an empty page.
  • The failure-streak counter (RecoveryError.Attempt) resets after each successful page — backoff measures consecutive failures, not lifetime calls (error, error, success, error yields Attempt values 1, 2, then 1 — not 1, 2, 3).
  • RecoveryComplete{Count} is emitted once, after the empty page, with the total recovered across all pages (still emitted with Count: 0 on a clean, empty run — F12 semantics preserved).
  • Context cancellation mid-pagination: emit RecoveryAbandoned{Attempts} and return without RecoveryComplete (current abandonment shape). Pages already forwarded are already durable rows; whatever isn't dispatched before shutdown is re-recovered at next start — the existing at-least-once contract, unchanged.
  • Keep the defensive chunking loop: a misbehaving Recoverer that returns more than limit still gets split into ≤batchSize units.

The documented F5 semantics (a failing Recoverer deliberately backs up the whole pipeline; broadcast drains work4a before live traffic) are untouched — the gate now opens after the final empty page instead of after the single big call.

3. Keyset pagination in the adapter (sqladapter/recovery.go)

type Recovery struct {
    handle      *sql.DB
    cursor      uint64 // advances past each successfully returned page; starts at MIN(id)-1 of the backlog
    boundary    uint64 // MAX(id) of the backlog, snapshotted on the first call
    snapshotted bool
}
  • First call only: snapshot the actual bounds of the backlog up front via 
    SELECT MIN(id), MAX(id) FROM Messages WHERE dispatched IS NULL;
    Both NULL (no backlog) means recovery is already complete — return the empty page without ever issuing a page query. Otherwise cursor = min - 1 (auto-increment ids start at 1, so no underflow) and boundary = max. Deriving both bounds from the backlog itself — rather than a table-wide MAX(id) — buys two things: the min bound lets every page query skip the potentially enormous prefix of old, already-dispatched rows via a primary-key range (no reliance on an index over dispatched), and the max bound stops pagination at the end of the actual backlog instead of walking the dispatched tail of the table.
  • Every subsequent call: 
    SELECT id, type, payload
  FROM Messages
 WHERE dispatched IS NULL AND id > ? AND id <= ?   -- cursor, boundary
 ORDER BY id
 LIMIT ?;                                           -- limit
  • Advance cursor to the last returned id only after the row scan completes cleanly (including rows.Err()) — a failed page is re-served on retry, never skipped.
  • NewRecovery(handle) is unchanged; the message construction (pooling note, hard-coded application/json) is unchanged.
Why the boundary snapshot matters: live traffic persists new dispatched IS NULL rows while recovery is paging (persistence runs concurrently; only broadcast waits on recovery). Without an upper bound, later pages would pick those rows up and recovery would publish them — and so would the live broadcast path when its turn comes, a systematic duplicate for everything written during the recovery window; under sustained traffic, pagination would chase the head and never terminate. Bounding pages to the backlog's MAX(id)-at-first-call reproduces today's single-snapshot semantics exactly: recovery owns the backlog that predates startup, the live path owns everything after.

Trade-offs and alternatives considered

AlternativeWhy not
Keep the signature; document loop-until-emptySilent semantic change — existing implementations compile and misbehave (see callout above). Rejected.
Iterator: Recover(ctx) iter.Seq2[*Message, error]Streams elegantly, but mid-stream failure handling gets murky: the pipeline's retry loop would need to abandon a partially consumed sequence and start a fresh one, re-yielding rows already forwarded (extra duplicate publishes on every transient error). Paging keeps retry trivially resumable. Also a heavier conceptual ask of third-party implementers than "return the next page."
Callback: Recover(ctx, emit func(...*Message) error) errorInversion of control complicates the station's select/abandonment logic and tests; same retry murkiness as the iterator.
Page-size state in the adapter (NewRecovery(handle, pageSize))Two knobs that must agree (adapter page vs. pipeline recoveryBatchSize); passing limit per call keeps one owner and zero new configuration surface.

Honest downsides of the chosen approach:

  • The Recoverer contract is more demanding. Implementations must maintain a cursor and a snapshot boundary correctly; a buggy one that re-serves rows causes duplicate publishes (tolerated by the at-least-once contract, but noisy). Mitigation: the contract doc spells out the rules, and the sqladapter implementation is the worked example. The pipeline does not add a non-advancing-page watchdog — that's complexity guarding against a contract violation the docs forbid; flagged here as a conscious omission.
  • Memory is bounded in rows, not bytes. 64 × 16 MB worst-case payloads is still ~1 GB per page in pathological cases. Realistic payloads are KBs, so the practical bound is MBs; a byte-budgeted page would require schema-aware sizing and is not worth it now. Noted for the future: recoveryBatchSize could become an Options knob without any further contract change.
  • Late-committing transactions holding an auto-increment id ≤ boundary but committing after recovery pages past it are invisible to recovery — identical to today's single-SELECT behavior (such rows belong to live units and are dispatched by the live path). No regression; documented for completeness.
