Proposal: F4 — abandonment must not strand (or falsely release) blocked callers

2026-06-12 · github.com/smarty/harness/v2 · follows the adversarial pre-release review, finding F4 · evidence drawn from smarty/messaging v3.17.2 and the billing-context prototype consumer (domain-transformation-phase-9-chunk-C)

Background

When the pipeline context is cancelled while the Writer is failing, persistence abandons the in-flight unit and deliberately does not forward it (internal/pipeline/03_persistence.go:42-44). The abandoned unit's completions are never invoked, so every goroutine parked in BlockingEntrypoint.Handle (waiter.Wait(), 00_entrypoint.go:71) blocks forever.

The non-negotiable constraint

The callers of Handle are messaging/v3 workers, and the messaging contract gives a handler no way to decline acknowledgment. Handler.Handle(ctx, ...any) returns nothing, and the streaming worker acks unconditionally the moment Handle returns — using the hard context, which is typically still alive during a soft shutdown:

// messaging/v3 streaming/worker.go:122-128
func (this *defaultWorker) deliverBatch() bool {
    if len(this.currentBatch) > 0 {
        this.handler.Handle(this.deliveryContext(), this.currentBatch...)
    }
    return this.stream.Acknowledge(this.hardContext, this.unacknowledged...) == nil
}
Therefore the review's parenthetical suggestion (in F4) — "run completions on abandonment" — is rejected. A unit is abandoned precisely because it was never durably stored. If Handle returned normally, the worker would ack and the message would be lost. The invariant we must preserve: a goroutine blocked in Handle for a non-stored unit must never return normally. That leaves exactly two outcomes for those goroutines: block forever, or die.

Why "block forever" (the status quo) wedges real deployments

Verified end-to-end in the billing-context prototype (main/api/application.go): pipeline.BlockingEntrypoint is handed to streaming.SubscriptionOptions.AddWorkers(...), and dominoes closes the message consumer before the pipeline listeners. The wedge chain on a SIGTERM-while-DB-is-down:

edgerunner cancels ctx + dominoes cascade begins→ persistence abandons unit (no completion)→ worker stuck in Handle→ subscriber.shutdown blocks on <-workersDone even after hardShutdown (subscriber.go:91-93)→ manager.Listen's WaitGroup never returns→ dominoes never reaches pipeline Close — process wedged until external SIGKILL

The messaging library's hard-shutdown escape hatch cannot help: a worker blocked inside handler.Handle never observes the hard context, and defaultSubscriber.shutdown waits unconditionally on workersDone after firing it.

Approach: fail loudly — panic the stranded callers

Make abandonment a first-class, observable outcome that propagates back to blocked callers as a panic. The panic guarantees the messaging worker never reaches its ack (its goroutine dies, taking the process with it — streaming has no recover), and it converts today's silent permanent hang into an immediate, explained crash at a moment when the process is already shutting down and can never make progress. The broker redelivers; at-least-once is preserved. This is consistent with the module's existing, documented serialization-panic policy.

Design

  1. Completion callbacks carry the outcome. In internal/pipeline/contracts.go, change: 
    batch struct {
    instructions []any
    complete     func(stored bool)   // was func()
}
unitOfWork struct {
    results     []*contracts.Message
    completions []func(stored bool)  // was []func()
}
    The completion stage (04_completion.go) invokes complete(true); it runs strictly after persistence forwards, so true always means "durably stored." (Broadcast abandonment needs no handling — it occurs after the ack stage, and recovery redispatches.)
  2. Persistence invokes completions with false on abandonment. In 03_persistence.go:42-44, before continue: 
    if !stored {
    for _, complete := range unit.completions {
        complete(false)
    }
    continue // not forwarded: no ack, no broadcast; MQ redelivers
}
    Once the context is cancelled, every subsequent queued unit takes the same path after a single failed Write attempt, so the input channel still drains and the stage-by-stage shutdown chain stays intact. The abandoned unit's pooled messages are deliberately not recycled (terminal never sees them) — the process is about to die; see trade-offs.
  3. The entrypoint propagates failure to its parked callers. prepare() threads a shared failed flag into the complete closure and now also returns it: 
    func (this *entrypoint) prepare(messages ...any) (waiter *sync.WaitGroup, item *batch, failed *atomic.Bool)
    Inside the closure, on !stored the flag is set and monitoring.BatchAbandoned{} tracked before waiter.Done(). (Revised during implementation: originally a plain *bool relying on the Done → Wait happens-before edge — correct, but it read as a race to humans; atomic.Bool makes the safety locally self-evident.) After waiter.Wait() returns, Handle panics with monitoring.ErrBatchAbandoned if the flag is set; await does the same, but only in its waiterDone branch — the two caller-departed branches remain byte-for-byte unchanged (they are protected by the 16k-iteration WaitGroup-misuse regression test). complete(false) still recycles the batch; the waiter need not be recycled before the panic (the process is dying — a one-line comment beats the pooled object).
  4. New sentinel + observation. In contracts/monitoring: 
    BatchAbandoned struct{}                  // tracked once per abandoned batch, before the panic
ErrBatchAbandoned = errors.New(
    "harness: batch abandoned before durable write; " +
    "panicking so the message broker never acknowledges and will redeliver")
    The panic value is ErrBatchAbandoned so any recovering layer can identify it with errors.Is.
  5. Documentation. contracts.Pipeline.BlockingEntrypoint doc gains: returns ⇒ durably stored; panics on shutdown-abandonment (and why). Package doc gains a shutdown-semantics paragraph. The review doc's F4 entry is updated to fixed with a verification note, and F6 gains a note that stranded-caller wedges no longer hold RLock forever.

Trade-offs and open questions

  • A library that panics. Deliberate and narrow: only when the durability contract is unsatisfiable (context cancelled + write never succeeded) — a state from which the process can never recover anyway. The alternative (block forever + "hosts must hard-kill" documentation) leaves the dominoes wedge in place for every host that waits on its workers, and makes F6's Close() deadlock permanent. The module already has a documented serialization-panic policy; this extends the same philosophy.
  • One panic kills all in-flight batches. Correct, not collateral: none of them were acked, all get redelivered.
  • Abandoned units leak pooled objects. Messages/waiters of abandoned batches are not recycled. Irrelevant — the first panic ends the process. Not worth code to "fix."
  • Duplicate deliveries increase. Inherent to at-least-once; consumers are already required to be idempotent (publish-then-mark dispatch has the same property).
  • Decided — await panics too (reviewer decision, 2026-06-12): uniformity with Handle, no false 200 for unsaved work, and both real HTTP stacks recover the panic (httpserver's HandlePanic(true), or net/http's per-connection recovery).
  • Not addressed here: F6's full-channel Close() deadlock (narrowed but not eliminated), F5 recovery-stall coupling. Separate findings, separate proposals.
