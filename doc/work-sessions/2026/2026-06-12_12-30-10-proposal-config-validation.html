Proposal: Configuration Validation — resolves F2 of the adversarial pre-release review

2026-06-12 (rev 2: error return instead of panic) · github.com/smarty/harness/v2 · research: adversarial pre-release review (F2, HIGH)

Background

The review found (F2, HIGH) that neither harness.New nor pipeline.Build validates any configuration value. The worst case is silent and total: Options.SerializerCount(0) builds a pipeline whose fan-in closes work2 immediately — downstream stages exit cleanly while the execution stage blocks forever on its first send to the never-consumed work1, and every BlockingEntrypoint.Handle caller hangs for the life of the process with no observation emitted. The full set of un-validated edges:

InputToday's effect
SerializerCount(0) (or negative)Deadlocked pipeline; callers hang forever, silently
BurstCapacity(0)admit() computes 0/0 = NaN; shedding never fires; every send rendezvouses unbuffered
Negative capacitiesmake(chan, n) panics deep inside Build with an unhelpful runtime message
ExecutionUnitSize(0) (or negative)Coalescing silently degrades to unit-per-batch
ShedThreshold outside [0, 1], or NaNUndefined shedding behavior (e.g. NaN never sheds; >1 never sheds; negative sheds everything)

This is a foundational library about to be consumed by other teams; a configuration that can never move a message must fail at construction, not cost a consumer an afternoon under load.

Approach

Fail fast with an error: New (and pipeline.Build) return (contracts.Pipeline, error). Exactly one of the two is non-nil. Construction happens at application startup, so the application detects the non-nil error and exits gracefully — log it, return non-zero, done. No partially constructed pipeline ever escapes: on any violation, Build returns the zero contracts.Pipeline and the error before any channel, pool, or goroutine plumbing is created.

// new signatures
func New(ctx context.Context, options ...option) (contracts.Pipeline, error)        // root package
func Build(ctx context.Context, config Configuration) (contracts.Pipeline, error)   // internal/pipeline

This is a breaking change to the v2 signature, made deliberately now — the module is unreleased (the whole point of this review cycle), and the only in-repo caller is config_test.go:89. An error return beats the alternatives:

  • vs. panic: an error lets the host application shut down on its own terms (flush logs, release resources, exit with a chosen status code) instead of unwinding through a stack trace. Idiomatic for constructors that take user-supplied configuration.
  • vs. silently clamping to defaults: hides the caller's mistake. A consumer who writes SerializerCount(0) meant something; running with a value they didn't choose is the same class of surprise we're trying to eliminate, just less severe.

Validation rules

FieldValid rangeNote
BurstCapacity≥ 10 breaks shedding math (NaN) and the documented buffering contract
PipelineBufferCapacity≥ 00 (unbuffered) is legal and meaningful; only negatives rejected
ExecutionUnitSize≥ 10 has no coherent meaning (coalesce nothing?)
SerializerCount≥ 10 is the deadlock; negatives panic later anyway
ShedThreshold0 ≤ v ≤ 1, not NaN0 remains valid (shed all HTTP traffic — useful for maintenance drain) but gets a doc-comment warning on the option; see below

Where the check lives

A config.validate() error method on pipeline.Configuration, called as the first statement of pipeline.Build. Placing it in the internal package (rather than in harness.New) means any future internal caller of Build is covered too, and the rule sits next to the struct it governs. harness.New simply propagates:

// config.go (root)
func New(ctx context.Context, options ...option) (contracts.Pipeline, error) {
    var cfg pipeline.Configuration
    for _, apply := range Options.defaults(options...) {
        apply(&cfg)
    }
    return pipeline.Build(ctx, cfg)
}

// internal/pipeline/pipeline.go
func Build(ctx context.Context, config Configuration) (contracts.Pipeline, error) {
    if err := config.validate(); err != nil {
        return contracts.Pipeline{}, err
    }
    ...
}

The error names the offending field, the received value, and the valid range — and reports all violations at once (collected and joined via errors.Join), so a consumer with two bad values fixes both in one iteration:

harness: invalid configuration: SerializerCount must be >= 1 (got 0)
harness: invalid configuration: ShedThreshold must be in [0, 1] (got 1.5)

A module-level sentinel contracts.ErrInvalidConfiguration wraps every violation, so callers can test errors.Is(err, contracts.ErrInvalidConfiguration) — consistent with the module's existing sentinel style (ErrPersistence, ErrBroadcast).

Documentation companion changes

  • New's doc comment documents the contract: exactly one of (pipeline, error) is non-nil; a non-nil error means the configuration is invalid and the application should not proceed.
  • Options.ShedThreshold doc comment gains: "A value of 0 sheds all HTTP traffic; values outside [0, 1] cause New to return an error."
  • Each tunable option's doc comment states its valid range, matching the table above.
  • The research document's F2 entry is flipped to data-state="fixed" with a verification note, and F14's "Degenerate configurations" gap is marked closed.

Trade-offs

  • Breaking signature change. Every existing call site (only config_test.go:89 in-repo, plus any unreleased consumers) must handle the error. Doing it before the v2 tag is precisely why this review ran first; after tagging, this door closes.
  • The error is ignorable. Unlike a panic, a careless caller can discard the error and use the zero contracts.Pipeline — its nil BlockingEntrypoint will nil-panic on first use, which is at least immediate and obvious rather than a silent deadlock. go vet/linters catch the unchecked error in practice.
  • ShedThreshold(0) stays legal. The review called it a footgun, but it has a legitimate use (drain mode) and rejecting it would conflate "suspicious" with "invalid." Documentation carries that load instead. Open to revisiting.
  • PipelineBufferCapacity(0) stays legal. Unbuffered stage channels are slow but correct. Same reasoning.
  • No validation of collaborator nil-ness. defaults() already guarantees non-nil collaborators through harness.New; a direct pipeline.Build caller passing nil collaborators is an internal-package concern out of scope here (only our own tests do that, and they pass non-nil).
