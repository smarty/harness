Architecture (as reviewed)

recovery→ work4a → entrypoint→ batches → execution→ work1 → serialization ×N (fan-out/fan-in)→ work2 → persistence→ work3 → completion (ack)→ work4b → broadcast→ work5 → terminal (recycle pools)

*contracts.Message and *unitOfWork objects are drawn from sync.Pools in execution and returned in terminal. Persistence and broadcast retry forever with capped exponential backoff + jitter until context cancellation. sqladapter implements Writer (multi-row INSERT, IDs derived from LAST_INSERT_ID() + stride), Dispatcher (publish then UPDATE … SET dispatched), and Recovery (SELECT … WHERE dispatched IS NULL).

CRITICAL F1 — Recycled pooled messages retain a stale Type; wrong type names are persisted to the database REPRODUCED01_execution.go:51-58 · 06_terminal.go:24-29 · sqladapter/writer.go:84-86

The execution stage takes a *contracts.Message from the pool and resets only Value and Content:

message := this.newMessage()
message.Value = result
message.Content.Reset()        // Type, ID, ContentType NOT cleared

The terminal stage returns messages to the pool with all fields intact. Meanwhile, sqladapter.Writer assigns Type only when it is empty:

if message.Type == "" {
    message.Type = this.typeNames[reflect.TypeOf(message.Value)]
}

So the first trip through the pipeline writes Type onto the pooled object; every subsequent reuse of that object for a different value type carries the previous type name into the INSERT. Recovered messages (which arrive with Type set from the DB) are also recycled into the same pool by terminal, seeding the pool with stale types from the very first moments of process life.

Reproduction (temporary test, since removed): a pipeline with one handler emitting 50 eventA then 50 eventB, single serializer, fake writer following the sqladapter pattern:

zz_repro_test.go:108: message 50: got Type="type-A" want "type-B"
zz_repro_test.go:118: distinct message pointers: 4, reused: 4, total writes: 100
zz_repro_test.go:120: 38 of 100 messages carried a stale pooled Type
--- FAIL: TestZZRepro_PooledMessageTypeStale

Only 4 distinct message objects served all 100 messages — pool reuse is the norm, not the exception. In production, rows in Messages would be persisted (and recovered/redispatched after a crash) under the wrong type name, which downstream consumers use to deserialize. This corrupts the durable record.

Why existing tests miss it: the race detector's sync.Pool instrumentation randomly drops Puts, and pipeline_test.go uses a Serializer/Writer that never reads or sets Type. The 100%-coverage figure measures statements, not field-lifecycle correctness.

Note: ID is equally stale on reuse but is unconditionally overwritten by Writer.assignIDs; ContentType is unconditionally set by serialization. Type is the only field guarded by an is-empty check. The reset point in execution (or the pool's recycle path) is the natural place to clear all identity fields.

HIGH F2 — No validation of Configuration: SerializerCount(0) silently builds a deadlocked pipeline BY INSPECTIONpipeline.go:25-74 · fanout.go:11-21 · config.go

None of the option values are validated. The worst case: Options.SerializerCount(0) produces a fan-out with zero workers; newFanIn with zero inputs closes work2 immediately, so persistence/completion/broadcast/terminal all exit cleanly — while the execution stage blocks forever on its first send to the never-consumed work1. Callers of BlockingEntrypoint.Handle then hang for the life of the process with no observation emitted. Other un-validated edges:

InputEffect
BurstCapacity(0)admit() computes 0/0 = NaN; NaN >= threshold is false, so admission never sheds, and every Handle send rendezvouses unbuffered
Negative capacitiesmake(chan, n) panics at Build (acceptable, but unfriendly)
ShedThreshold(0)Sheds all HTTP traffic (0/1024 ≥ 0) — possibly intended, but a footgun next to the documented range [0, 1]
ExecutionUnitSize(0)Coalescing condition len(completions) < 0 never true — degrades to unit-per-batch (harmless but undocumented)

Since this is a foundational library about to be consumed by other teams, New/Build silently accepting a configuration that can never move a message is the kind of hole that costs someone an afternoon.

HIGH F3 — Default SerializerCount=4 destroys cross-unit ordering between execution and persistence BY INSPECTIONfanout.go:35-46 · pipeline.go:47

Units of work are fanned out across 4 serialization goroutines and fanned back in nondeterministically (fanIn merges with a WaitGroup over independent range loops). Two units executed in order A, B can arrive at persistence as B, A — so database id order (the stream order seen by recovery and by any consumer reading the Messages table) does not reflect execution order whenever more than one unit is in flight.

For a store-and-forward event pipeline this is a significant semantic: if any consumer assumes "ascending id = causal order" (a natural assumption given the auto-increment ID scheme and the ordered recovery query), the default configuration violates it under load. The integration tests assert ordering only with strictly sequential blocking calls (one unit in flight), so this is invisible to the suite.

Within a single unit, message order is preserved (the serializers iterate unit.results in order, and persistence appends in order). The hazard is exclusively between units. Either the ordering contract should be documented as "per-unit only," or serializer fan-out needs an order-preserving fan-in — worth an explicit decision before tagging, since it shapes consumer expectations forever.

MEDIUM F4 — Persistence abandonment strands blocked Handle callers forever BY INSPECTION03_persistence.go:40-46 · 00_entrypoint.go:71

When the writer keeps failing and the context is cancelled, persistence abandons the unit and deliberately does not forward it (continue — correct for the no-ack/redelivery contract). But the dropped unit's completions are never invoked, and entrypoint.Handle intentionally ignores context cancellation (waiter.Wait() with no escape — see TestHandle_IgnoresContextCancel). Result: every MQ-delivery goroutine parked in Handle at abandonment time blocks forever. If the host process waits on those goroutines during shutdown, shutdown wedges; the operator's only recourse is a hard kill.

Likely tolerable if consumers are expected to die anyway after cancelling the context, but the contract "cancel the context to unwind" is only half-true today: it unwinds the pipeline stages, not the callers. Deserves either a fix (run completions on abandonment — the unit was not acked durably, but the caller contract is "blocks until durably stored," which can no longer ever happen) or a loud doc note on BlockingEntrypoint.

MEDIUM F5 — A failing Recoverer at startup throttles the live pipeline to a standstill BY INSPECTION05_broadcast.go:41-45 · recovery.go:49-61

broadcast.Listen fully drains the recovery channel (work4a) before touching live traffic (work4b). Recovery retries its query forever with backoff. If the database is unreachable at startup (the very scenario recovery exists for), work4a stays open, broadcast never starts on work4b, and after ~(2×buffer+2) units the whole pipeline backs up: completion blocks sending to work4b, then persistence blocks, then callers block. Live writes that could succeed (e.g., recovery DB ≠ writer DB, or a transient SELECT failure) are stalled behind recovery's retry loop.

In the common deployment (same DB for both) this is mostly moot — writes would fail too. But the coupling is invisible to operators: the symptoms are "persistence is fine, dispatching stopped" with only RecoveryError observations as the clue. Worth documenting, and note that RecoveryError wraps the raw error without a sentinel like ErrRecovery (inconsistent with persistence/broadcast — see F12).

MEDIUM F6 — Close() can deadlock against a Handle blocked on a full work channel BY INSPECTION00_entrypoint.go:59-73, 128-137

Handle holds the read lock across the channel send (this.work <- item), which is what makes close-vs-send safe. The flip side: if the work channel is full because downstream is wedged (see F4/F5, or a dispatcher outage with an uncancelled context), Handle blocks holding RLock, and Close() blocks forever acquiring Lock. Shutdown ordering then never reaches the stages that would unwedge things. The same applies to await's enqueue path only until its context is cancelled (it selects on ctx.Done()), so HTTP callers self-rescue; MQ-style Handle callers do not.

This is the corollary of the (well-tested) design decision that Handle ignores cancellation. The triple {full channel, wedged downstream, Close} is plausible during an incident — exactly when an operator tries a graceful restart.

MEDIUM F7 — Recovery loads the entire undispatched backlog into memory in one query BY INSPECTIONsqladapter/recovery.go:22-58 · pipeline/recovery.go:40-47

The recovery SELECT has no LIMIT and accumulates every row (full payloads) into a slice before the pipeline chunks it (recoveryBatchSize=64 chunks the output, not the read). After an extended dispatcher outage — precisely when backlog is largest — startup memory is O(backlog). With mediumblob payloads up to 16MB each, a few thousand stuck rows can be gigabytes. A crash-loop here is self-reinforcing: OOM at startup prevents the dispatch that would shrink the backlog. A LIMIT + loop-until-empty (or streaming rows into units) would bound it; today nothing does.

LOW F8 — initialContentBufferSize allocates 1024 bytes of content, not capacity BY INSPECTIONpipeline.go:38-40 
return &contracts.Message{Content: bytes.NewBuffer(make([]byte, initialContentBufferSize))}

bytes.NewBuffer treats its argument as initial contents: a fresh pooled message starts with 1024 zero bytes readable in Content. Execution's Content.Reset() hides this today, so it is latent rather than live — but any future code path that reads a fresh message's Content before reset (or a Writer that handles a never-serialized message) sees 1KB of zeros. The intent is plainly make([]byte, 0, initialContentBufferSize).

LOW F9 — Routing silently no-ops when the prefixed method and the generic Execute/Apply disagree BY INSPECTIONscanner.go:8-48 · routing.go:35-57

The scanner uses ExecuteXxx/ApplyXxx method signatures only to build the type→handler map; dispatch then calls the handler's generic Execute(any, func(...any)). Two silent failure modes follow:

  • A type with an ExecuteRenewal(Renewal, func(...any)) method whose generic Execute switch forgets the Renewal case: the message routes, then vanishes without a trace (no observation, no panic).
  • A prefixed method whose message parameter is an interface type: the map key is the interface type, but lookup uses reflect.TypeOf(message) (always concrete) — never matches, silently dropped.
  • A type with prefixed methods that forgets to implement the generic interface at all is skipped entirely by scan (the v.(executor) assertion) — again silently.

The double bookkeeping (prefixed method for discovery + hand-written type switch for dispatch) is an inconsistency-by-construction; at minimum the build step could fail loudly when a registered type has discoverable methods but no generic interface, or when types route to nothing.

LOW F10 — sqladapter Writer/Dispatcher are stateful and not safe for concurrent use, undocumented BY INSPECTIONsqladapter/writer.go:18-26 · sqladapter/dispatcher.go:17-22

Both reuse instance-level statement/args buffers across calls. The pipeline invokes each from exactly one goroutine, so this is correct in situ — but they are exported types in a public module with no doc comment saying "single goroutine only." A consumer who shares one Writer across two pipelines (or calls it directly) gets silently interleaved SQL. One sentence of doc (like the pipeline's own router comment, which does this well) closes it.

LOW F11 — Dispatcher UPDATE ignores rows-affected; ID=0 messages mark nothing BY INSPECTIONsqladapter/dispatcher.go:56-59

The UPDATE … WHERE dispatched IS NULL AND id IN (…) result is discarded. If a message arrives with ID=0 (a Writer that failed to assign, or a non-sqladapter Writer), the publish succeeds but the row is never marked — recovery republishes it on every restart, forever. The Writer side carefully validates affected and first; the Dispatcher's symmetric check is absent. (Re-running over already-marked rows is correctly idempotent thanks to the dispatched IS NULL guard; this is only about rows that should have matched and didn't.)

LOW F12 — Monitoring inconsistencies: no ErrRecovery sentinel; no RecoveryAbandoned/recovered-count observation BY INSPECTIONcontracts/monitoring · pipeline/recovery.go:55-58

Persistence and broadcast wrap errors in ErrPersistence/ErrBroadcast and emit …Abandoned{Attempts} on context cancellation. Recovery emits RecoveryError with the raw error (no sentinel) and abandons silently (returns nil with no observation). An operator alerting on "abandoned" observations misses the recovery case; an error-unwrap chain looking for module sentinels misses recovery errors. Also, nothing reports how many messages were recovered — useful operational signal that is currently invisible.

LOW F13 — Pooled slice growth is retained forever; one giant batch permanently inflates pool memory BY INSPECTION01_execution.go:42-58 · 03_persistence.go:30 · sqladapter buffers

unitOfWork.results/completions are truncated with [:0] but never re-capped, so a single pathological batch (e.g., one command broadcasting 100k events) grows pooled slices that sync.Pool then retains across GC cycles (Go pools survive one GC). The same pattern applies to persistence.buffer, and the sqladapter statement/args buffers. Likewise the multi-row INSERT has no batch-size cap: a huge unit produces a single statement that can exceed MySQL's max_allowed_packet, which then retries forever (it is deterministic, not transient) — a poison-message loop. Low likelihood, but the failure mode is an unkillable retry loop with PersistenceError spam.

INFO F15 — Minor inconsistencies and dead weight various
  • router.Apply (routing.go:29-33) is never called by the pipeline — only by its own tests. Either it is future API (then say so) or it is dead code shipping in v2.0.0.
  • contracts.Monitor is invoked concurrently from entrypoint, serialization (×N), persistence, broadcast, and recovery goroutines, but the interface doc does not state that implementations must be safe for concurrent use. The test fixtures all lock; consumers must too.
  • Recovery (internal/pipeline) is the only exported stage type in an internal package — harmless, but inconsistent with the "every internal stage type is unexported" claim in the package doc of config.go.
  • generic.Drain is exported (within internal/) but unused by the pipeline — test-only utility living in prod code.
  • Recovery hard-codes ContentType: "application/json" with a clear TODO-style comment — fine for now, but it will silently mislabel payloads the moment a non-JSON serializer is configured. The comment acknowledges this; flagging so it makes the schema-migration backlog.
INFO F16 — Things checked that are fine (so you don't have to re-check)
  • Close-vs-send race: Handle/await hold RLock across the send; Close takes Lock before close(work) — no send-on-closed-channel possible. Idempotent Close tested.
  • Departed-caller waiter pooling: the deliberate non-Put of the waiter on in-flight departure (entrypoint.go:101-107) is correct and protected by an 16k-iteration stress test that trips the runtime's WaitGroup misuse detector if regressed. Nice work.
  • backoff(): attempt 1 → 0.5–1s, capped at 15–30s; rand.N scope always positive; no overflow (cap at 6 doublings).
  • Writer panic safety: recover() in Write converts legacyWrite panics into rollback + error; the named-return error path rolls back on every failure; commit errors propagate. DB tests cover panic, rollback, and stride paths.
  • Shutdown drain: stage-by-stage defer close(output) chain means a single entrypoint.Close() drains everything in order; verified by inspection and by TestZeroOptionsPipelineRunsInertly + integration shutdown tests.
  • At-least-once dispatch: publish-then-mark with dispatched IS NULL guard means a crash between the two yields a duplicate publish, never a lost message — consistent with the store-and-forward contract.
  • Race detector: full suite (including DB integration) clean under -race.
  • Serialization panic policy: deliberate, documented in three places, tracked before panicking. Agree with the design.
