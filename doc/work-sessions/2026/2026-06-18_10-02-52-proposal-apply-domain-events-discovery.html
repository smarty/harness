Discover replayable domain events by scanning the Domain

Delete snapshots.LoadOptions.DomainEvents(events ...any) with no replacement option: the already-required RegisteredEvents registry doubles as the replay opt-in. When it is provided, Load reflectively discovers which event types the Domain can Apply — reusing the method-shape machinery currently buried in internal/pipeline/scanner.go — resolves them to canonical names, and hands storage a plain []string (no type registry crosses the storage boundary).

Background

The new snapshots package (load.go, save.go, contracts.go) replaces the older snapshots.go. Its Load entry point loads a snapshot, applies it to a caller-supplied Domain, then — optionally — replays every event newer than the snapshot's high watermark through the same Domain.Apply method.

Today the caller signals which events to replay by handing Load a slice of sample instances: 

// current: load.go
// TODO: scan the domain type to discover which events it can Apply.
func (loading) DomainEvents(events ...any) loadOption {
    return func(config *loadConfig) { config.DomainEvents = events }
}

This is redundant and error-prone. The Domain already declares exactly which events it understands — it has one Apply<Foo>(Foo) method per event type. Forcing the caller to also pass a parallel list of sample instances invites three failure modes:

  • Drift — the caller adds an ApplyOrderShipped method but forgets to add orderShipped{} to the DomainEvents(...) list, so that event silently never replays.
  • Over-listing — the caller passes an event the Domain has no handler for; it loads from storage and is applied to a no-op, wasting a query and muddying intent.
  • Ceremony — every call site repeats a list the type system could derive.

The harness already solved this exact problem for the pipeline. internal/pipeline/scanner.go reflectively discovers Execute<Foo> / Apply<Foo> methods and routes by the parameter's reflect.Type. We should lift that method-shape detection into a shared spot and reuse it here.

Goal

Remove DomainEvents outright and add no new option in its place. The RegisteredEvents option is already required whenever events are replayed — it carries the type↔name maps needed to query for and deserialize them. So its mere presence is the signal: provide RegisteredEvents and Load discovers + replays; omit it and Load stops at the snapshot. The event set itself is derived from the Domain by reflection, never hand-listed. 

// RegisteredEvents indexes the event types that can be (de)serialized.
// Providing it ALSO opts Load into replaying events since the loaded
// snapshot's high watermark: Load scans the Domain for Apply<Foo>(Foo)
// methods, resolves each to its canonical name via namesByType, and
// loads those events. Omit it (the default) to skip replay entirely.
func (loading) RegisteredEvents(typesByName map[string]reflect.Type, namesByType map[reflect.Type]string) loadOption {
    return func(config *loadConfig) {
        config.EventRegistry.TypesByName = typesByName
        config.EventRegistry.NamesByType = namesByType
    }
}

Net effect on the option surface: DomainEvents is deleted, no ApplyDomainEventsSinceHighWatermark is introduced, and RegisteredEvents gains a second, documented responsibility. Load's options shrink to Logger, Storage, SnapshotID, Domain, RegisteredEvents, LoadedSnapshot.

Resolved decisions (from review) D0. No boolean opt-in option. The presence of RegisteredEvents (a non-empty NamesByType) is what enables event replay; DomainEvents is removed.
D1. A type the Domain can Apply but that is absent from RegisteredEvents is a configuration error — Load returns an error (never silently skips).
D2. domainscan.AppliedTypes de-duplicates its result.
D3. The shared package is named internal/domainscan.
D4. Load resolves the discovered types to their canonical string names and passes a []string directly on the storage operation. storage.LoadEventsSince drops its TypeNames map entirely and replaces Events []any with Types []string — so storage no longer consults any type registry. This requires changes to the storage operation and the MySQL mapper.

Approach

1 · Extract the method-shape machinery into internal/domainscan

internal/pipeline/scanner.go owns three reflective primitives that are not pipeline-specific: isApplyMethod, isExecuteMethod, and handledType (plus the broadcastFuncType sentinel). They encode the single source of truth for "what does a discoverable handler method look like?"

We lift these into a new leaf package, internal/domainscan, as exported helpers, and add one new convenience for the snapshots use case: 

// internal/domainscan/scan.go
package domainscan

func IsApplyMethod(method reflect.Method) bool     { /* moved verbatim */ }
func IsExecuteMethod(method reflect.Method) bool   { /* moved verbatim */ }
func HandledType(method reflect.Method) reflect.Type { /* moved verbatim */ }

// AppliedTypes returns the de-duplicated message types discoverable on v
// via methods shaped like Apply<Foo>(Foo). Returns nil for a nil value
// or one with no such methods.
func AppliedTypes(v any) (results []reflect.Type) {
    if v == nil {
        return nil
    }
    seen := make(map[reflect.Type]bool)
    t := reflect.TypeOf(v)
    for x := range t.NumMethod() {
        method := t.Method(x)
        if !IsApplyMethod(method) {
            continue
        }
        if handled := HandledType(method); !seen[handled] {
            seen[handled] = true
            results = append(results, handled)
        }
    }
    return results
}

The naming matters: internal/pipeline/scanner.go already defines a function literally named scan, so the package cannot be called scan (the import name would collide with the function). domainscan avoids the collision and reads well at both call sites: domainscan.IsApplyMethod(...), domainscan.AppliedTypes(...).

2 · Refactor pipeline/scanner.go to consume the shared helpers

The pipeline keeps its own scan(types...) (it maps to pipeline's executor/applicator instances, which is pipeline-specific) and validateDomainTypes, but they now delegate the method-shape checks to domainscan. This is a pure refactor — no behavior change — guarded by the existing pipeline tests (routing, config validation), which stay green. 

// after: internal/pipeline/scanner.go (excerpt)
for x := range t.NumMethod() {
    method := t.Method(x)
    if isExecutor && domainscan.IsExecuteMethod(method) {
        executors[domainscan.HandledType(method)] = append(...)
    }
    if isApplicator && domainscan.IsApplyMethod(method) {
        applicators[domainscan.HandledType(method)] = append(...)
    }
}

3 · Make the storage operation registry-free — it takes []string

The mapper's only use of Events + TypeNames today is to turn each sample instance into a canonical name for the WHERE type IN (...) clause. If Load resolves those names itself (it owns the registry anyway), the storage operation reduces to "here are the type names to query for" — no reflection, no registry, nothing for storage to look up: 

// internal/storage/operations.go
LoadEventsSince struct {
    HighWatermark uint64
    Events        []any
    TypeNames     map[reflect.Type]string
    Types         []string  // canonical stored type names to query for
    Result        struct { ... }
}
// storage/mysql/mapper.go — loadEventsSince (excerpt)
for e, event := range op.Events {
    name := op.TypeNames[reflect.TypeOf(event)]
for e, name := range op.Types {
    if slices.Contains(statement.args, any(name)) {
        continue
    }
    statement.args = append(statement.args, name)
    ...
}

With TypeNames gone, internal/storage/operations.go no longer imports reflect for this operation. Resolution moves up into Load (section 5), which is the only place that holds the registry.

5 · Derive & validate the event set inside load()

The replay branch changes from "did the caller pass events?" to "did the caller supply a registry, and does the Domain declare any applicable types?" Per D1, every discovered type must be present in the RegisteredEvents registry; an applicable-but-unregistered type is a configuration error, not a skip: 

if len(config.EventRegistry.NamesByType) == 0 {
    return result, nil // no registry → caller doesn't want replay
}
applied := domainscan.AppliedTypes(config.Domain) // de-duplicated
if len(applied) == 0 {
    return result, nil // registry provided, but the Domain applies no events — nothing to replay
}
var typeNames []string
for _, t := range applied {
    name, ok := config.EventRegistry.NamesByType[t]
    if !ok {
        return result, fmt.Errorf("%w: %s", errUnregisteredEventType, t) // D1: fail loud
    }
    typeNames = append(typeNames, name)
}
loadEvents := &storage.LoadEventsSince{
    HighWatermark: loadedSnapshotResult.HighWatermark,
    Types:         typeNames, // []string — storage needs no registry
}
// ... unmarshal each loaded row via TypesByName, then Domain.Apply — unchanged

A new sentinel joins contracts.go: errUnregisteredEventType = errors.New("snapshots: domain applies an unregistered event type").

Why RegisteredEvents stays — and now lives entirely in snapshots Both registry maps are still needed, but now they are consumed only inside the snapshots package: NamesByType turns each discovered reflect.Type into the canonical name composed into the []string query, and TypesByName turns each loaded row's stored name back into a concrete type to unmarshal and Apply. The Domain knows the types; the caller knows the stored names; storage now knows neither. So RegisteredEvents(typesByName, namesByType) is unchanged, but the registry no longer crosses the storage boundary.
