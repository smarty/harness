Capped Exponential Backoff with Jitter for Pipeline Retries

2026-06-11 — github.com/smarty/harness

Background

Two TODOs remain in the repository, both marking the same gap:

Location Retry loop Current behavior
internal/pipeline/03_persistence.go:59persistence.store()flat 1s wait between attempts
internal/pipeline/05_broadcast.go:72 broadcast.dispatch()flat 1s wait between attempts

Both loops retry forever (by design — until context cancellation). A flat 1-second delay hammers a struggling dependency (database, message broker) and, with many harness instances, produces synchronized retry storms. Exponential backoff eases pressure; jitter de-synchronizes the fleet; and a cap ensures the delay never grows so long that a recovered dependency sits idle while the pipeline stays clogged waiting out a huge sleep.

Note: recovery.go:58 contains an identical flat-1s retry loop without a TODO. For consistency, this proposal applies the same backoff there too (cheap once the helper exists). Easy to drop if you'd rather keep scope to the two TODOs.

Approach

Add a small, pure, deterministic-given-its-inputs backoff function next to the existing wait helper in internal/pipeline/retry.go, then replace time.Second at the three call sites with backoff(attempt). No contract changes; contracts.Waiter stays as-is, and the existing test fixtures (which record the durations passed to their injected wait) keep working — assertions just become range-based.

Algorithm: capped exponential with equal jitter

// backoff returns the delay before the next retry: exponential growth from
// base, capped at max, with "equal jitter" (uniform in [d/2, d)) to
// de-synchronize concurrent retriers while guaranteeing a floor.
func backoff(attempt int) time.Duration
ParameterValue Rationale
base 1 * time.Second matches current behavior on first retry
cap 30 * time.Secondretries resume regularly; pipeline never stalls > 30s past recovery
growth ×2 per attemptstandard; reaches the cap by attempt 6
jitter equal jitter: d/2 + rand(d/2)bounded below (unlike full jitter), so retry cadence stays predictable

Resulting delay windows:

AttemptUncappedJittered range
1 1s [0.5s, 1s)
2 2s [1s, 2s)
3 4s [2s, 4s)
4 8s [4s, 8s)
5 16s [8s, 16s)
6+ 30s (cap)[15s, 30s)

Implementation notes:

  • Use math/rand/v2 (rand.Int64N) — no seeding ceremony, concurrency-safe.
  • Guard the shift against overflow: compute min(cap, base<<(attempt-1)) by clamping the shift count (attempt ≥ 6 ⇒ already at cap; attempt > ~32 must not wrap).
  • Call sites change from this.wait(this.ctx, time.Second) to this.wait(this.ctx, backoff(attempt)) — the TODO comments are deleted.

Trade-offs

  • Unexported, hard-coded constants. Base/cap are not configurable. This matches the repo's style (the flat 1s was hard-coded too) and avoids contract churn. If operators later need tuning, the function signature can grow without touching contracts.
  • Tests become range assertions. Because jitter is random, fixtures assert each recorded wait falls within its [d/2, d) window rather than should.Equal, time.Second. Alternative — injecting a rand source — buys exactness at the cost of another constructor parameter through three stages; not worth it.
  • Equal jitter vs. full jitter. Full jitter (uniform in [0, d)) spreads load slightly better in huge fleets but can produce near-zero delays, defeating the back-off. Equal jitter's floor fits the "retry regularly but don't hammer" goal here.
  • Recovery scope creep. Including recovery.go goes beyond the two TODOs. It's the same pattern and the same one-line fix; flagging it so you can veto.

Where: internal/pipeline/retry.go

Signature:

func backoff(attempt int) time.Duration

Behavior to satisfy (the tests in retry_test.go will check all of these):

  • Attempt 1 yields a duration in [500ms, 1s); attempt n doubles the window up to the cap.
  • Attempt 6 and beyond always yields a duration in [15s, 30s) — including absurd values like attempt = 1<<30 (no overflow, no negative durations).
  • Attempt ≤ 0 is treated as attempt 1 (defensive; the loops start at 1).
  • Repeated calls with the same attempt produce varying values (jitter is real, not constant).

Hints: math/rand/v2's rand.Int64N(n) returns uniform [0, n); clamp the shift count before shifting (time.Duration is an int64 of nanoseconds — 1s << 35 overflows). Remember house style: no blank lines at function start/end, name the return result if you use a named return.
