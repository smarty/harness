Route the sqladapter.Writer through storage.DB

The Mapper owns the transaction and the deprecated legacyWrite hook — so *sql.Tx never crosses the storage boundary, and there is nothing to abstract.
Proposal · 2026-06-16 13:43:16 · revised · branch master · author Michael Whatcott
Status: DRAFT — awaiting review. No code to be written until approved.

Background

We are converging every sqladapter type onto a single seam: storage.DB (Handle(ctx, operation any) error), implemented by mysql.Mapper. The Dispatcher already routes through it via storage.MarkMessagesDispatched, and the Recovery type is a straightforward follow-on. The Writer is the hard case, and it is the subject of this proposal.

The Writer is the only type that opens an explicit transaction. Reading what is actually inside that transaction (writer.go:83–93) is the key to the whole plan:

tx := handle.BeginTx(...) INSERT INTO Messages (...) VALUES (...), (...), ... ← derives IDs from this one Result legacyWrite(ctx, tx, values...) ← deprecated, caller-supplied, *needs the tx* tx.Commit() ← (Rollback on error / panic)
Core insight The transaction does not exist for the INSERT. ID derivation reads RowsAffected() and LastInsertId() off the same sql.Result the INSERT returns — there is no second statement and therefore no cross-statement state to protect. The transaction exists for one reason only: to make the legacyWrite hook atomic with the INSERT. Remove the hook and the transaction has nothing left to wrap.

We confirmed we cannot retire legacyWrite yet (migration of downstream callers is incomplete). So the obstacle is not "the transaction"; it is specifically the hook's signature, func(ctx, *sql.Tx, …any), which hands callers a raw *sql.Tx.

The realization that simplifies everything The hook is inherently *sql.Tx-coupled and MySQL-specific. Its natural home is therefore the MySQL reference adapter — mysql.Mapper — which already imports database/sql and already owns the real transaction. If the hook lives there, *sql.Tx never has to cross the storage boundary at all, so there is nothing to abstract: no executor interface, no bridge, no type assertion. An earlier draft threaded the hook through the operation behind a storage.Execer shim; that machinery is deleted in this revision.

Approach

Three changes, each smaller than the last. The transaction and the hook both migrate into the Mapper as private implementation details; storage stays purely data; the Writer becomes a one-line shim.

1 — A pure-data storage operation

storage.InsertMessages carries only the batch. No callback, no database/sql import — the storage package remains fully driver-agnostic, exactly like MarkMessagesDispatched. 

// internal/storage/operations.go
type InsertMessages struct {
    Messages []*contracts.Message // Mapper assigns IDs back onto these pointers
}

2 — The Mapper owns the transaction and the hook (chainable setter)

The deprecated hook is registered through a chainable setter, keeping it off the primary constructor so the mainline is already in its post-deprecation end-state. stride moves here too — it must match the server's auto_increment_increment, so it is database-topology configuration that belongs with the type that owns the connection. 

// internal/storage/mysql/mapper.go
type Mapper struct {
    handle      *sql.DB
    stride      uint64
    statements  *generic.PoolT[*statement]
    legacyWrite func(context.Context, *sql.Tx, ...any) // Deprecated; never nil — no-op default
}

func NewMapper(handle *sql.DB, stride uint64) *Mapper {
    return &Mapper{
        handle:      handle,
        stride:      cmp.Or(stride, 1),
        statements:  generic.NewPoolT(newStatement),
        legacyWrite: func(context.Context, *sql.Tx, ...any) {}, // no-op until overridden
    }
}

// Deprecated: WithLegacyWrite registers the transitional hook run inside the same
// transaction as the INSERT, and returns the Mapper for chaining. Retained only
// for migration; omit it for new callers.
func (this *Mapper) WithLegacyWrite(fn func(context.Context, *sql.Tx, ...any)) *Mapper {
    this.legacyWrite = fn
    return this
}

// usage:  mapper := mysql.NewMapper(handle, stride).WithLegacyWrite(hook)

A new case *storage.InsertMessages in Handle absorbs the INSERT, the ID derivation (assignIDs, errRowsAffected, errIdentityFailure), and the transaction with its panic-to-rollback discipline. The hook is always invoked — a no-op by default — so there is no per-call nil-check and exactly one code path. This matches the current Writer, which already opens a transaction on every batch. 

func (this *Mapper) insertMessages(ctx context.Context, op *storage.InsertMessages) (err error) {
    tx, err := this.handle.BeginTx(ctx, nil)
    if err != nil { return err }
    defer func() {                          // preserves the Writer's panic⇒rollback contract
        if r := recover(); r != nil { _ = tx.Rollback(); err = fmt.Errorf("panic during write: %v", r) }
        else if err != nil { _ = tx.Rollback() }
    }()
    if err := this.insert(ctx, tx, op.Messages); err != nil { return err }
    this.legacyWrite(ctx, tx, values(op.Messages)...) // always called; no-op by default
    return tx.Commit()
}

insert(ctx, tx, messages) runs the multi-row INSERT and ID derivation against the transaction. values(messages) projects each message.Value into the ...any the hook expects. Because every batch now runs in a transaction, no shared executor abstraction is needed.

3 — The Writer becomes a one-line shim

The Writer stops touching *sql.DB, stride, transactions, and the hook entirely. It builds the operation and hands it to storage.DB — nothing else. 

type Writer struct { db storage.DB }

func NewWriter(db storage.DB) *Writer { return &Writer{db: db} }

func (this *Writer) Write(ctx context.Context, messages ...*contracts.Message) error {
    if len(messages) == 0 { return nil }
    return this.db.Handle(ctx, &storage.InsertMessages{Messages: messages})
}

Data flow

sqladapter (thin shim) │ storage (pure data) │ mysql (owns DB + tx + hook) ────────────────────────────────│ ─────────────────────────────│ ────────────────────────────────── Writer.Write(messages) │ │ db.Handle(ctx, op) ─────────┼─► InsertMessages{Messages} ─┼─► Mapper.insertMessages │ │ BeginTx │ │ INSERT → assignIDs │ │ legacyWrite(ctx, tx, values…) ← no-op by default │ │ Commit / Rollback-on-panic no *sql.Tx here no database/sql here *sql.Tx CONTAINED here ↑

Trade-offs & open questions

ConcernAssessment
The Mapper now carries a deprecated, insert-only concern Mitigated by the chainable WithLegacyWrite setter: it is opt-in and clearly Deprecated:, so the primary NewMapper(handle, stride) path is already in its end-state. When the hook is finally removed, delete WithLegacyWrite, the legacyWrite field, and the Mapper's transaction branch — the plain INSERT remains. Nothing in storage changes, because nothing legacy ever lived there.
Value derivation (message.Value) lands in the Mapper The hook's ...any are the message Values; with the hook on the Mapper, the Mapper projects them from op.Messages. That is a sliver of "legacy receives the Values" knowledge in the MySQL adapter — acceptable, since the adapter is the hook's owner. (Rejected alternative: a redundant Values []any field on the operation, which would pollute the op with a legacy-only field.)
Always-tx with a no-op hook default The hook defaults to a no-op and is always called, so there is no per-call nil-check and exactly one code path (this also lets us drop the shared executor abstraction). The cost: every batch opens a transaction even when no legacy work runs — but the current Writer already does so, so it is not a regression. The no-transaction fast path for new callers is deferred until legacyWrite is removed (see Out of scope).
Public API change NewWriter(handle, stride, legacy) → NewWriter(db storage.DB); NewMapper(handle) → NewMapper(handle, stride), with the deprecated hook chained on via .WithLegacyWrite(...). The hook now configures the Mapper, not the Writer. sqladapter is the reference adapter ("copy and adapt"); external callers rewire construction. Documented in both package docs.

Out of scope

The Recovery type's migration onto storage.DB is a natural sequel but is not part of this proposal. Removing legacyWrite outright (and with it WithLegacyWrite, the Mapper's legacyWrite field, the no-op default, and its always-on transaction) is the eventual cleanup this design is shaped to make trivial — at which point the INSERT can drop to a single transaction-free statement. Also out of scope here.
