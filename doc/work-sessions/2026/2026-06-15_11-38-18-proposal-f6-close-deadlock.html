Proposal: F6 — Close() must not deadlock against a Handle blocked on a full work channel

2026-06-15 · github.com/smarty/harness/v2 · resolves the adversarial pre-release review, finding F6 (MEDIUM, open) · builds directly on the F4 abandonment-panic contract

Background

The entrypoint protects its work channel with an *sync.RWMutex. Senders (Handle, await) hold the read lock across the channel send; Close takes the write lock before close(this.work). This is what makes close-vs-send safe — the mutual exclusion guarantees no goroutine is mid-send when the channel is closed, so the runtime can never panic with "send on closed channel."

The flip side is the deadlock F6 describes. Handle holds RLock across the send (00_entrypoint.go:67-75):

func (this *entrypoint) Handle(_ context.Context, messages ...any) {
    this.lock.RLock()
    if this.closed {
        this.lock.RUnlock()
        return
    }
    waiter, item, failed := this.prepare(messages...)
    this.work <- item          // <-- blocks here while holding RLock
    this.lock.RUnlock()
    ...

If the work channel is full because the downstream pipeline is wedged (a dispatcher/writer outage with an uncancelled context), the send blocks forever holding RLock. Close() then blocks forever acquiring the write lock, and — because Go's RWMutex gives a pending writer priority — every subsequent Handle/admit piles up behind it too. Graceful shutdown never reaches the stages that would unwedge things; the operator's only recourse is SIGKILL.

downstream wedged (dispatcher outage, ctx alive)→ work channel fills→ Handle blocks on send, holding RLock→ Close() blocks on Lock()→ shutdown never reaches downstream stages — process wedged until SIGKILL

What the F4 fix did and did not close

The F4 fix narrowed F6: persistence abandonment now fails completions and the parked caller panics, so a caller stranded by a cancelled-context-plus-failing-writer no longer holds RLock forever. But that only rescues the cancelled-context path. The remaining window is exactly the one above: full work channel + wedged downstream + uncancelled context (e.g., a dispatcher outage during normal operation, then an operator attempts a graceful restart). The send is blocked on a full channel — it never even reaches persistence, so the F4 completion-failure mechanism never fires.

Latent twin in await. await selects its send against ctx.Done(), so HTTP callers normally self-rescue when their request context cancels. But a caller whose context never cancels during the wedge (e.g. context.Background(), a long-lived stream) holds RLock the same way and deadlocks Close identically. The review flagged Handle; this proposal closes both for symmetry.

Approach: give a blocked send a shutdown escape via the entrypoint's own done channel

The entrypoint already owns a done chan struct{} that Close closes to release Listen(). We make the blocked send selectable against that same signal, and we close done before acquiring the write lock so a blocked sender can observe it and release RLock — letting Close proceed to acquire the lock and close the channel.

A sender released this way never durably stored its work (it never even entered the pipeline). By the contract established in F4, it must not return normally — that would let a message broker acknowledge unstored work. So the escape ends the same way F4 does: panic with monitoring.ErrBatchAbandoned (after tracking monitoring.BatchAbandoned). The process dies, the broker redelivers, at-least-once holds. F6's resolution reuses F4's sentinel and contract verbatim — no new vocabulary.

Design

  1. Close signals before it locks, and is idempotent via sync.Once. Closing done must happen outside the write lock (a blocked sender holds RLock, so Close cannot acquire Lock until the sender releases it — which only happens once it observes done). sync.Once replaces the current if !this.closed idempotency guard, since close(done) now lives outside the lock and double-close panics: 
    func (this *entrypoint) Close() error {
    this.closeOnce.Do(func() {
        close(this.done)     // (1) release any sender blocked on a full work channel
        this.lock.Lock()     // (2) now reachable: blocked senders have let go of RLock
        this.closed = true
        close(this.work)     // (3) safe: no sender holds RLock at this instant
        this.lock.Unlock()
    })
    return nil
}
  2. Handle selects the send against done, preferring an available slot. A naive single select { send; done } would, once done is closed, pick randomly between sending and escaping even when the channel has room — spuriously panicking callers during an otherwise-clean shutdown. A two-step select prefers a ready send deterministically and only becomes escapable when the send would actually block: 
    select {
case this.work <- item:
    // fast path: slot available, take it
default:
    select {
    case this.work <- item:           // normal full-channel backpressure...
    case <-this.done:                  // ...now escapable at shutdown
        this.lock.RUnlock()           // release BEFORE panicking, or Close re-deadlocks
        this.abandon(waiter, item)
        this.waiters.Put(waiter)      // safe: abandon() drove the count to 0; no detached Wait()
        this.monitor.Track(batchAbandoned)
        panic(monitoring.ErrBatchAbandoned)
    }
}
this.lock.RUnlock()
this.monitor.Track(batchInFlight)
...
    During normal operation done is never ready, so the inner select blocks on the send exactly as today — full-channel backpressure is unchanged. The escape fires only after Close has closed done.
  3. await gains the same done case. Its select already has this.work <- batch and <-ctx.Done(); we add a third <-this.done case (and the same prefer-send fast path). The ctx.Done() branch is unchanged (track callerDeparted, return normally — the caller departed, nothing was stored, no false success to suppress). The new done branch mirrors Handle: abandon, track batchAbandoned, panic. HTTP stacks recover the panic into a 5xx, exactly as they already do for the F4 failed.Load() panic.
  4. No new monitoring or contract types. monitoring.BatchAbandoned and monitoring.ErrBatchAbandoned already exist from F4. The done escape simply tracks/panics with them. abandon() already exists and does exactly what we need (waiter.Done() + recycle the batch).

Why close(work) stays safe

The invariant is unchanged: no goroutine executes this.work <- item after close(this.work).

  • A sender holds RLock throughout both the outer and inner select. Close cannot acquire Lock while any sender holds RLock, so at the instant close(work) runs, no sender is inside a send. ✓
  • After close(work), closed == true (set under the same Lock), so any new Handle/await takes the early return before reaching a send. ✓
  • In the brief window between close(done) and Close acquiring Lock, a new sender may still pass the (not-yet-set) closed check and enter its select. With done already closed it either sends into a still-open channel (room available — fine) or escapes via done (panics — correct, it is shutdown and the work is unstored). Either way: no send after close. ✓
Close: close(done)→ blocked sender observes done, RUnlocks, panics→ Close acquires Lock→ close(work)→ downstream drains & stages close in order

Trade-offs and open questions

  • A graceful shutdown under saturation panics in-flight senders. If the work channel is genuinely full at Close time, blocked senders escape-and-panic rather than waiting for a drain. This is inherent to the lock structure: close(work) (which begins the drain) cannot run until blocked senders release RLock, so a blocked sender can never "wait for the drain." The prefer-send fast path ensures senders with an available slot still succeed, so this only bites a truly-full channel. The outcome — process dies, broker redelivers unstored work — is correct under at-least-once and consistent with F4's "violent but honest" philosophy. The alternative (status quo) is a permanent deadlock requiring SIGKILL, which is strictly worse.
  • Rejected: silent return on the done escape. Returning normally would let a broker ack work that was never stored — reintroducing the exact bug F4 closed. The panic is non-negotiable for the Handle (MQ) path.
  • Rejected: an active-sender WaitGroup that Close drains. A counter incremented per send and waited on by Close would also let Close avoid the lock-vs-send standoff, but it is more machinery for the same outcome and still cannot let a blocked sender complete a send into a wedged downstream. Reusing the already-present done channel is the smaller, more legible change.
  • Rejected: a send timeout. Any fixed timeout is arbitrary and turns a wedge into silent data loss or spurious panics under mere slowness. The done signal is precise: escape exactly when (and only when) shutdown is requested.
  • await's exposure is more theoretical than Handle's. Real HTTP request contexts cancel, so await usually self-rescues today. We fix it anyway for symmetry and to cover non-cancelling contexts; the cost is one extra select case.
  • Decided — await's done escape panics (reviewer decision, 2026-06-15): consistent with its existing failed.Load() panic and with Handle. At shutdown the work is unstored, so a false 200 is worse than the 5xx an HTTP stack produces by recovering the panic. The quieter callerDeparted-style return was rejected for that reason. (The unchanged ctx.Done() branch still returns normally — that is a genuine caller departure, not an abandonment.)
