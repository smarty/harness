Skip Empty Work

Reject zero-instruction batches at the door · forward empty units without an empty write/dispatch  ·  branch mikewhat/more-counters

Background

The branch-scan of more-counters surfaced two related observations. The new throughput counters now emit a Count: 0 event whenever an empty unit flows through persistence or broadcast, and the two tests that pinned the "empty results trigger an empty write/dispatch" behavior were deleted. That raised the underlying question: why do we do empty work at all?

There are two distinct ways "nothing to do" reaches the pipeline today:

SourceWhat happens nowWhy it is wasteful
Empty batch
Handle(ctx) with no messages		 A batch with instructions: [] is pooled, enqueued onto work, coalesced into a unit, and the caller blocks on the waiter until an empty write "completes." Occupies a work slot, consumes admission capacity, and makes a blocking caller wait on a durable write that stores nothing.
Empty unit
a real batch whose instructions produce zero result messages		 persistence calls writer.Write() with an empty buffer; broadcast calls dispatcher.Dispatch() with an empty buffer. Both now also emit a Count: 0 counter. An empty INSERT / empty publish is a pointless round-trip (and a retry-able failure surface) for a no-op, plus counter noise.

Note these are independent: even with empty batches rejected, a non-empty batch can still legitimately produce an empty unit (an Execute that emits nothing, or a coalesced unit where no instruction yielded output). So both fixes are needed and neither subsumes the other.

Approach

1 · Reject empty batches at the entrypoint

The only path that can mint a zero-instruction batch is entrypoint.Handle(ctx, messages...) (the BlockingEntrypoint). The shedding path routes through httpAdapter.Handle, which calls await(ctx, message) once per message — structurally always exactly one instruction, never zero. So a single guard at the top of Handle is sufficient and correct:

func (this *entrypoint) Handle(ctx context.Context, messages ...any) {
+	if len(messages) == 0 {
+		return // nothing to store; never enters the pipeline.
+	}
	this.lock.RLock()
	if this.closed {
		...

Returning immediately is consistent with the BlockingEntrypoint contract ("returns only after a durable write"): there is nothing to write, so there is nothing to wait for. No batch is pooled, no waiter is created, and crucially no abandonment panic can arise — there is no in-flight work to abandon.

2 · Forward empty units without writing / dispatching

In both persistence and broadcast, short-circuit the store/dispatch when the unit carries no result messages, but still forward the unit so its completions fire downstream (this is what unblocks any waiting caller) and pooled values are reclaimed at the terminal stage.

persistence (03_persistence.go):

func (this *persistence) Listen() {
	defer close(this.output)
	for unit := range this.input {
+		if len(unit.results) == 0 {
+			this.output <- unit // nothing to persist; already "stored" vacuously.
+			continue
+		}
		for _, message := range unit.results {
			this.buffer = append(this.buffer, message)
		}
		stored := this.store()
		...
		this.monitor.Track(monitoring.ResultsPersisted{Count: len(unit.results)})
		this.output <- unit
	}
}

broadcast (05_broadcast.go), symmetric, inside processFrom:

func (this *broadcast) processFrom(input chan *unitOfWork) {
	for unit := range input {
+		if len(unit.results) == 0 {
+			this.output <- unit // nothing to dispatch; forward (already durable).
+			continue
+		}
		for _, message := range unit.results {
			this.buffer = append(this.buffer, message)
		}
		this.dispatch()
		this.buffer = this.buffer[:0]
		this.output <- unit
	}
}
Buffer hygiene is preserved. Both buffers are reset (persistence via generic.Reclaim after store(), broadcast via [:0] after dispatch()) at the end of every non-empty iteration, so an empty unit's early continue simply leaves a length-zero buffer untouched. The single-goroutine-per-stage invariant is unchanged.

Resulting flow

Trade-offs & open questions

  • Counter semantics. Empty units will no longer emit ResultsPersisted{Count:0} / ResultsDispatched{Count:0} — which resolves the noise from branch-scan finding #2. If anyone wanted those zero events as a "unit traversed the stage" heartbeat, this removes them. I believe that is not the intent (the counters are throughput, not traversal), but worth confirming.
  • Silent rejection of empty batches. Handle(ctx) with no messages now returns with no observable trace. Open question: do we want a monitoring event (e.g. an EmptyBatchIgnored counter) to make the no-op visible, or is silent-and-free preferable? Default proposal: silent, since it is a caller mistake with no consequence.
  • Lost test coverage is restored, inverted. The deleted TestEmptyResultsTriggersEmptyWrite/Dispatch tested the old behavior. We replace them with tests asserting the new behavior (skip + forward), so the empty path is pinned again — just to the opposite contract.
  • No change to durability invariants. Empty units forward exactly as before (they were always forwarded); we only drop the empty I/O call in between. Nothing that was acknowledged-after-store changes ordering or acknowledgement.
