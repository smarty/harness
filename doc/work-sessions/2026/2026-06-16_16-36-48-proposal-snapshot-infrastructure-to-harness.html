Background

billing-context carries five pieces of code tagged // TODO: will eventually be moved to an infrastructure package. They are domain-agnostic snapshot and event-replay plumbing that happens to live in the application repo. The goal is to generalize them and move them into github.com/smarty/harness/v2 so every store-and-forward context can reuse them.

Two facts drive the whole design:

  • Operations live in internal/storage. Per your direction, the three operation structs join the existing pipeline operations in internal/storage/operations.go and are handled by the existing exported storage/mysql.Mapper.Handle switch. Because internal/ is not importable from another module, billing-context can no longer name these types directly — it reaches them only through the exported snapshots package (below). This is the lever that forces the generalization rather than a copy-paste.
  • The "reader" is the Mapper. InitializeDomain stops taking a bespoke storage.Reader and instead takes contracts.DB (Handle(ctx, operation any) error) — the same seam Writer, Dispatcher, and Recovery already route through. mysql.Mapper already implements it.

What "generalize" concretely means, per the three current TODO sub-notes ("table name will come from config", "type constraint will come from config"):

  • The snapshot operations (#3, #4) gain a TableName field so the query is no longer hard-wired to BillingSnapshots. The harness test schema already ships a generic Snapshots table.
  • The events operation (#5) drops the hard-coded type LIKE 'billing:%' OR type IN ('accounting:…') predicate. It instead carries a variadic set of sample event objects plus a map[reflect.Type]string type-names map; the Mapper resolves those to canonical names and builds the type IN (…) clause.

1 — Three new operations in internal/storage/operations.go

These join InsertMessages, MarkMessagesDispatched, etc. They follow the established convention: the caller passes a pointer and the Mapper populates result fields. None import database/sql, so the package stays driver-agnostic. 

// internal/storage/operations.go (additions)

// SaveSnapshot persists one domain snapshot row into TableName.
type SaveSnapshot struct {
    TableName       string
    Timestamp       time.Time
    HighWatermark   uint64
    Payload         []byte
    ContentType     string
    ContentEncoding string
}

// LoadLatestSnapshot loads the most recent snapshot row from TableName.
type LoadLatestSnapshot struct {
    TableName string
    Result    LoadedSnapshot // populated by the handler
}
type LoadedSnapshot struct {
    Found           bool
    HighWatermark   uint64
    Payload         []byte
    ContentType     string
    ContentEncoding string
}

// LoadEventsSince loads serialized events newer than HighWatermark whose stored
// type matches one of EventTypes, each resolved to its canonical name via TypeNames.
type LoadEventsSince struct {
    HighWatermark uint64
    EventTypes    []any                      // sample instances; key into TypeNames
    TypeNames     map[reflect.Type]string // reflect.Type → canonical name
    Result        struct {
        NewHighWatermark uint64
        Events           []Event
    } // populated by the handler
}
type Event struct {
    Type    string
    Payload []byte
}

Per your Q2 answer, LoadEventsSince carries the objects + map and the Mapper resolves names at query-build time (rather than a constructor pre-resolving to []string).

2 — Three new Handle cases in storage/mysql/mapper.go

All driver detail stays in the package that already owns database/sql. Table names cannot be bound as ? placeholders, so TableName is interpolated into the SQL string — guarded by a strict identifier check (see the warning below). The event-name list is bound positionally. 

func (this *Mapper) Handle(ctx context.Context, operation any) error {
    switch op := operation.(type) {
    // … existing cases …
    case *storage.SaveSnapshot:
        return this.saveSnapshot(ctx, op)
    case *storage.LoadLatestSnapshot:
        return this.loadLatestSnapshot(ctx, op)
    case *storage.LoadEventsSince:
        return this.loadEventsSince(ctx, op)
    default:
        return storage.ErrUnsupportedOperation
    }
}
saveSnapshot: INSERT INTO <TableName> (created, high_watermark, payload, content_type, content_encoding) VALUES (?, ?, ?, ?, ?) loadLatestSnapshot: SELECT high_watermark, payload, content_type, content_encoding FROM <TableName> ORDER BY id DESC LIMIT 1 (Found=false when no row) loadEventsSince: SELECT id, type, payload FROM Messages WHERE type IN (?, ?, …) AND id > ? ORDER BY id ASC

Note: SaveSnapshot takes SaveSnapshot.Timestamp as the created value, matching billing's current behavior of stamping the time at the call site rather than using NOW(). loadEventsSince reads from the standard Messages table the rest of the Mapper already targets (see Trade-offs for why it does not also take a TableName).

SQL identifier safety Interpolating TableName into a query is injection-prone if it ever carries untrusted input. Even though table names here come from trusted config/code, the Mapper will validate TableName against ^[A-Za-z0-9_]+$ and return an error otherwise, and wrap it in backticks. An empty TableName is rejected rather than silently producing FROM .

3 — New exported snapshots package

This is the entire public surface. It owns its own tiny collaborator interfaces (per your direction) and constructs the now-internal operations on the caller's behalf, dispatching them through contracts.DB. Because it lives under the module root, it is allowed to import internal/storage. 

package snapshots

type Logger interface{ Printf(string, ...any) }
type Applicator interface{ Apply(message any) }

type DomainInitializationReport struct {
    PreviousHighWatermark uint64
    NewHighWatermark      uint64
    EventsAppliedCount    uint64
    Error                 error
}

// InitializeDomain loads the latest snapshot from snapshotTable, applies it, then
// loads and applies every event newer than the snapshot's high watermark.
func InitializeDomain[S any](
    ctx context.Context,
    logger Logger,
    db contracts.DB,
    snapshotTable string,
    messageTypes map[string]reflect.Type, // name → type, decodes events
    typeNames map[reflect.Type]string,    // type → name, filters the IN clause
    domain Applicator,
    events ...any,                       // sample instances of the event types to replay
) (result DomainInitializationReport)

// LoadSnapshot decompresses (if gzip) and unmarshals a snapshot payload into S.
func LoadSnapshot[S any](
    payload []byte, contentEncoding string, highWatermark uint64, logger Logger,
) (snapshot S, err error)

// SaveSnapshot persists one snapshot row (thin wrapper over db.Handle so external
// modules need not name the internal operation type).
func SaveSnapshot(
    ctx context.Context, db contracts.DB, tableName string,
    timestamp time.Time, highWatermark uint64, payload []byte, contentType, contentEncoding string,
) error

// LoadEventsSince loads, decodes, and returns events newer than highWatermark, for
// callers (snapshot-inspect, snapshot-to-sqlite) that replay against a chosen snapshot.
func LoadEventsSince(
    ctx context.Context, db contracts.DB, highWatermark uint64,
    messageTypes map[string]reflect.Type, typeNames map[reflect.Type]string, events ...any,
) (decoded []any, newHighWatermark uint64, err error)
Why SaveSnapshot and LoadEventsSince become functions too Your item list names only InitializeDomain and LoadSnapshot as functions, with #3/#4/#5 as mapper-handled operations. But once those operation structs are internal, billing-context (a separate module) cannot construct them to hand to db.Handle. The snapshots package therefore exposes thin wrappers as the public face of #3 and #5. These also remove the duplicated reflect/decode loops currently copy-pasted across snapshot-inspect and snapshot-to-sqlite. Flag me if you'd rather expose the operation structs directly from a non-internal package instead.

Trade-offs & open questions

  • The LIKE 'billing:%' shortcut is gone. Billing relied on a prefix match plus a short explicit list. The generalized form requires callers to pass the full set of event Go types. This is more verbose but explicit and self-documenting, and it is the only domain-agnostic option. Billing's all.Messages registry can enumerate them.
  • LoadEventsSince has no TableName. You scoped TableName to the snapshot operations; events are read from the conventional Messages table that the rest of the Mapper already targets. If a context stores events elsewhere this needs revisiting — easy to add later. Confirm this is acceptable.
  • Two mappers over one *sql.DB. Billing keeps its own mapper for its many domain-specific reads; for snapshots/events it now constructs a harness/v2/storage/mysql.Mapper over the same *sql.DB and passes it as contracts.DB. Both are single-goroutine drivers, so coexistence is fine. (The billing migration is out of scope here — see below.)
  • LoadSnapshot signature changed from struct to primitives. Taking (payload, contentEncoding, highWatermark) rather than the internal LoadedSnapshot keeps that result type internal while letting external callers (who scan a row by hand) still call it. The alternative — exporting a LoadedSnapshot DTO — is also viable; say the word.
  • JSON v1 vs v2. LoadSnapshot in billing uses encoding/json; snapshot/main.go marshals with encoding/json/v2 behind a build tag. The moved LoadSnapshot will use stdlib encoding/json (v1) for decode, which reads v2-written JSON fine. Confirm harness should not take on the jsonv2 goexperiment dependency.
  • Scope of this proposal = harness only. The checklist below adds and tests the code in harness/v2. Actually rewiring billing-context's four call sites to consume it requires a tagged harness release and is tracked as a follow-up, not implemented here.

Your turn — loadEventsSince in the Mapper

The piece reserved for you This is the semantic heart of the whole task: turning a variadic of Go event objects into the type IN (…) filter. It is small, pure-ish, and has crisp test cases. Phase C's RED tests are written to drive it.

Implement, on storage/mysql.Mapper:

func (this *Mapper) loadEventsSince(ctx context.Context, operation *storage.LoadEventsSince) error

Behavior to build:

  • For each value in operation.EventTypes, resolve its canonical name via operation.TypeNames[reflect.TypeOf(value)]. Collect the names (dedupe is a nice touch).
  • If no names resolve, return without querying and leave Result empty — never emit a malformed IN ().
  • Build SELECT id, type, payload FROM Messages WHERE type IN (?, …) AND id > ? ORDER BY id ASC, binding one ? per resolved name plus operation.HighWatermark last. Reuse the pooled statement buffer the way markMessagesDispatched does (this.statements.Get() / reset() / defer Put).
  • Scan each row into storage.Event{Type, Payload}, append to operation.Result.Events, and track operation.Result.NewHighWatermark = id as you go. Return rows.Err() at the end.

Tests that must pass: TestLoadEventsSinceFiltersByTypeAndWatermark and TestLoadEventsSinceEmptyTypeSet (Phase C). Reference implementations to model the SQL/scan shape: markMessagesDispatched (IN-list building) and loadUndispatchedPage (row scan loop) in the same file.
