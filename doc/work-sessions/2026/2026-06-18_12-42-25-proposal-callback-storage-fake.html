Callback-Driven Storage Fake for the snapshots tests

Proposal (rev. 3, as built) · 2026-06-18 · package github.com/smarty/harness/v2/snapshots

This document reflects the implemented design. It started (rev. 2) by mirroring the sales-context/code/http/load_taxable_location_test.go pattern with a type-keyed callback registry embedded on the fixture. During implementation that was simplified: the registry is now an ordered slice of callbacks consumed in sequence (no reflect.Type map, no operation instance passed to prepareExec), and the fake is a plain field on the fixture rather than an embedded type. The sections below describe that final shape.

Background

The two test suites in snapshots/load_test.go and snapshots/save_test.go each drive their subject through a hand-rolled test double that implements storage.Storage (a single Exec(ctx, operation any) error method). Today both doubles use a type switch plus a bag of pre-seeded data fields (latest, byID, events, snapshotErr, capturedID, capturedQuery, …). The test's intent is smeared across three places: the struct's pre-seeded fields, the shared switch, and the assertions after the call. The double answers every operation type even when a test exercises one, and adding an operation means editing a switch shared by unrelated tests.

The reference pattern

load_taxable_location_test.go demonstrates the target shape. The fixture implements the dependency interface directly and holds a map[reflect.Type]callback; each test calls prepareRead to register the behavior for exactly the operation it cares about: 

type GetTaxableLocationShellFixture struct {
    *gunit.Fixture
    reads map[reflect.Type]any
}

func (this *…Fixture) Read(ctx context.Context, queries ...any) {
    for _, query := range queries {
        if callback, ok := this.reads[reflect.TypeOf(query)]; ok {
            reflect.ValueOf(callback).Call([]reflect.Value{reflect.ValueOf(query)})
        }
    }
}

func (this *…Fixture) prepareRead(query any, callback func(any)) {
    this.reads[reflect.TypeOf(query)] = callback
}

// …in a test:
this.prepareRead(new(storage.LoadCurrentTaxableLocation), func(a any) {
    query := a.(*storage.LoadCurrentTaxableLocation) // type-assert the intended op
    this.So(query.AccountID, should.Equal, uint64(111)) // verify inputs
    query.Result.Address = storage.TaxableLocation{…}     // set results
})

We keep the essence of this pattern — a fixture that registers per-call behavior as a callback that type-asserts the operation, checks inputs, and sets results — with two adaptations. First, storage.Storage.Exec returns an error, so our callback is func(any) error and dictates the returned error. Second, since Load/Save issue their Exec calls in a fixed sequence, we drop the reflect.Type key entirely: callbacks are stored in a slice and consumed in order, so prepareExec needs no operation instance.

Goal

Fold the storage double into each gunit fixture as a field. Delete loadStorageStub and saveStorageSpy. Each test appends, in call order, a callback that type-asserts the operation, verifies its input fields, populates its Result, and returns the error the case requires. The fake carries no per-operation knowledge.

The shared fake

Both suites live in one package and already share domain fixtures (domainState, marshalJSON, the applicator spies). Define one small helper in the shared region of load_test.go and hold it as a field on each fixture. The field is named db (not storage, which would shadow the imported package); a test passes &this.db wherever a storage.Storage is wanted. 

// fakeStorage is a storage.Storage test double whose Exec calls are answered, in
// registration order, by the callbacks a test adds via prepareExec. Each callback
// type-asserts the operation it expects, checks inputs, fills in its Result, and
// returns the error the case requires.
type fakeStorage struct {
    callbacks []func(operation any) error
    calls     int
}

func (this *fakeStorage) Exec(_ context.Context, operation any) error {
    callback := this.callbacks[this.calls]
    this.calls++
    return callback(operation)
}

func (this *fakeStorage) prepareExec(callback func(operation any) error) {
    this.callbacks = append(this.callbacks, callback)
}

calls serves double duty: it indexes the next callback and counts how many Exec calls happened (used by the skip/no-store assertions below). gunit hands each test a fresh fixture, so the slice and counter reset automatically — no Setup needed. An unexpected extra Exec (more calls than callbacks) indexes past the slice and panics, failing the test loudly.

Fixtures with the fake as a field

type LoadFixture struct {
    *gunit.Fixture
    db     fakeStorage // pass &this.db as the storage.Storage
    logged []string    // still the logger via Printf
}

type SaveFixture struct {
    *gunit.Fixture
    db fakeStorage
}

How the ordered slice fits Load's two calls

Save always issues one Exec (a SaveSnapshot). Load issues one or two: the snapshot load always, and a LoadEventsSince only when replay is enabled and the domain applies events. Because the sequence is fixed, an ordered slice fits exactly — a two-call test registers the snapshot callback first and the events callback second:

The reference has no "this query was skipped" assertion, but three of our load tests do. A skip test registers only the snapshot callback and then asserts this.So(this.db.calls, should.Equal, 1) — proving the events query never ran (had it run, the slice would be exhausted and Exec would panic). The calls counter is the one piece beyond the bare reference pattern.

How the tests read

Single call — verify input, set result

before

db := &loadStorageStub{
  byID: storage.LoadedSnapshotResult{
    Found: true, HighWatermark: 5,
    Payload: marshalJSON(…),
  },
}
… Load(Storage(db), SnapshotID(5)) …
this.So(db.capturedID, should.Equal, uint64(5))

after

this.db.prepareExec(func(a any) error {
  load := a.(*storage.LoadSnapshot)
  this.So(load.ID, should.Equal, uint64(5))
  load.Result = storage.LoadedSnapshotResult{
    Found: true, HighWatermark: 5,
    Payload: marshalJSON(…),
  }
  return nil
})
… Load(Storage(&this.db), SnapshotID(5)) …

Storage(&this.db) — the fixture's field is the double. The capturedID field disappears; its check moves into the callback, right where the operation arrives. The callback's type assertion (a.(*storage.LoadSnapshot)) documents which ordered call it answers.

Two calls — register callbacks in call order

this.db.prepareExec(func(a any) error { // 1st Exec: the snapshot load
    load := a.(*storage.LoadLatestSnapshot)
    load.Result = storage.LoadedSnapshotResult{
        Found: true, HighWatermark: 3, Payload: marshalJSON(domainState{Name: "snap", Count: 1}),
    }
    return nil
})
this.db.prepareExec(func(a any) error { // 2nd Exec: the events query
    query := a.(*storage.LoadEventsSince)
    this.So(query.HighWatermark, should.Equal, uint64(3)) // verify input field
    query.Result.Events = []storage.Event{
        {Type: "event:alpha", Payload: marshalJSON(eventAlpha{Order: 11})},
        {Type: "event:beta", Payload: marshalJSON(eventBeta{Order: 22})},
    }
    query.Result.NewHighWatermark = 7
    return nil
})

Error case & "never called"

// storage error: the callback just returns it
this.db.prepareExec(func(any) error { return boom })

// events query must be skipped: register only the snapshot callback, then…
this.So(this.db.calls, should.Equal, 1)

Trade-offs & notes

ConsiderationAssessment
Per-test callbacks are more verbose than pre-seeding struct fields. Accepted, and the point: each test tells its whole story in one place, with no shared switch to reason around. Matches the established sales-context convention.
The fake is a field (db) the fixture passes as &this.db, rather than an embedded type. The fixture is already the logger (via Printf); a field keeps the storage role explicit at the call site (Storage(&this.db)) instead of hiding it behind method promotion.
Callbacks run in registration order, so a test's registrations must match Load's call sequence (snapshot, then events). Accepted. The sequence is fixed and short, so order is easy to get right; in exchange the fake drops the reflect.Type map and prepareExec needs no operation instance. The callback's a.(*storage.X) assertion still names the operation it expects.
Terse type assertion op := a.(*storage.X) panics on a mismatch. Adopted from the reference (no comma-ok). A wrong-type panic is an immediate, located test failure — a coding error in the test itself.
An unexpected extra Exec (more calls than callbacks) panics with a raw index-out-of-range. Accepted for simplicity — it still fails the test loudly and points at Exec. A guard with a friendlier message could be added if it ever proves confusing.

Post-implementation refinement. The checklist above records the path actually taken, which built the fake type-keyed and embedded first. A follow-up pass then simplified it to the final shape described at the top of this document: the reflect.Type map became an ordered callbacks slice with a calls counter (so prepareExec no longer takes an operation instance, and len(this.executed) assertions became this.db.calls), and the embedded fakeStorage became a db field passed as &this.db. make test remained green throughout; snapshots coverage held at 99.0%.

Out of scope

No production code changes. storage.Storage, Load, and Save are untouched. The storage/mysql tests and their Mapper doubles are a separate concern and are not modified here.
