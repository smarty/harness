Discover @@auto_increment_increment at runtime

Stop trusting a hard-coded stride argument — let the MySQL Mapper discover it lazily on its first write, riding the pipeline's existing retry-forever model instead of blocking boot.

Topic: storage/mysql ID-derivation stride  ·  Date: 2026-06-22  ·  Scope: storage/mysql/mapper.go, its tests, README.md, config.go doc comment
Proposal only — no production or test code is written until you approve.

Background

The Mapper derives every message's primary key arithmetically from a single multi-row INSERT. It reads LAST_INSERT_ID() for the first row, then computes the rest: 

// storage/mysql/mapper.go:168 — assignIDs(...)
message.ID = uint64(first) + uint64(i)*this.stride

This is only correct when stride exactly matches the server's @@auto_increment_increment. Today that value is supplied by the caller as a constructor argument that silently defaults to 1: 

// storage/mysql/mapper.go:33
func NewMapper(handle *sql.DB, stride uint64, snapshotsTableName, messagesTableName string) *Mapper {
    return &Mapper{
        handle: handle,
        stride: cmp.Or(stride, 1), // silently defaults to 1
        ...
    }
}

// README.md:132 and README.md:231 — callers are told to hard-code it
mapper := mysql.NewMapper(db, 1, "Snapshots", "Messages")
The foot-gun. If the operator deploys against a MySQL server whose auto_increment_increment is not 1 (multi-source replication and Galera/Group-Replication clusters commonly set it to 2 or higher), the hard-coded 1 is silently wrong. Every derived ID after the first in a batch collides with or skips real rows — corrupting the monotonic-ID invariant the whole pipeline depends on. Nothing fails loudly.

Notably, the test suite already knows the right answer — it queries the server and feeds the result in, rather than trusting a literal (mapper_test.go:45): 

func (this *MapperFixture) autoIncrementIncrement() uint64 {
    var result uint64
    err := this.handle.QueryRow(`SELECT @@auto_increment_increment`).Scan(&result)
    this.So(err, should.BeNil)
    return result
}

The goal is to move that one query out of the test fixture and into the production Mapper, so the value is discovered automatically and the error-prone constructor argument disappears.

Why not at construction / boot

Your concern, and it's the right one. If the Mapper performed a synchronous SELECT @@auto_increment_increment at construction and returned an error when the DB is unreachable, the service couldn't finish initializing while MySQL is down. It would fail its infrastructure health/readiness check inside the allotted threshold and never reach a running state — turning a transient database outage into a service that can't even start.

Crucially, this dependency does not exist today. Callers build the handle with sql.Open(...), which is lazy — it validates the DSN but opens no connection. So constructing the Mapper performs zero network I/O, and the process can boot with MySQL down. The pipeline's documented invariant then takes over:

"persistence, broadcast, and recovery retry forever (backoff caps at 30s) until the pipeline ctx is cancelled." — CLAUDE.md, pipeline invariants

A blocking startup query would regress that resilience. So we must discover the stride in a way that participates in the existing retry-forever model rather than gating boot on it.

Approach recommended

Detect the stride lazily, on the Mapper's first write, and let any failure flow through the persistence stage's existing infinite-retry loop. Construction stays pure (no I/O, no error), so boot is never blocked. The stride parameter is removed entirely.

  • The stride field becomes an atomic.Uint64; 0 means "not yet discovered."
  • An ensureStride(ctx) guard runs at the top of the insert path: if the stride is still 0, it issues the one SELECT @@auto_increment_increment and caches the result.
  • If that query fails (DB down), ensureStride returns the error, the insert returns the error, and the persistence stage retries with backoff — exactly as it already does for a failed INSERT. No new retry machinery, no new failure mode.
  • Once cached, the stride is never re-queried; subsequent inserts read the atomic and skip straight to the arithmetic.
// ensureStride: detect once, lazily, concurrency-safe. 0 == not yet known.
func (this *Mapper) ensureStride(ctx context.Context) error {
    if this.stride.Load() != 0 {
        return nil
    }
    var detected uint64
    err := this.handle.QueryRowContext(ctx, `SELECT @@auto_increment_increment`).Scan(&detected)
    if err != nil {
        return err // → insert fails → persistence retries forever (existing behavior)
    }
    this.stride.Store(cmp.Or(detected, 1)) // guard against a pathological 0
    return nil
}

// insertMessages calls ensureStride before opening the transaction.
func (this *Mapper) insertMessages(ctx context.Context, op *storage.InsertMessages) (err error) {
    if len(op.Messages) == 0 {
        return nil
    }
    if err := this.ensureStride(ctx); err != nil {
        return err
    }
    // ... existing BeginTx / insert / assignIDs path, with assignIDs reading this.stride.Load()
}

The constructor stays non-fallible and I/O-free, so the two existing tests that build with a nil *sql.DB (mapper_assign_ids_test.go:23, mapper_invalid_table_test.go:28) keep working — they just drop the stride argument. The assign-IDs test, which needs a known stride to check the arithmetic, sets it directly via white-box access (subject.stride.Store(7)), since it lives in package mysql.

Trade-offs & open questions

ConcernAssessment
Detection deferred to first write, not boot. This is the point — see above. The stride is only ever used on the insert path, so detecting it there is both sufficient and the natural home for the retry. Reads (recovery, dispatch-marking) never need it.
Breaking API change. The signature drops one parameter. The only call sites are the README and the tests — blast radius is tiny and entirely inside this /v2 module.
Mutable state on a concurrent-safe type. The Mapper is documented as safe for concurrent use. atomic.Uint64 preserves that. In practice only the single persistence goroutine inserts, but the atomic keeps the contract honest.
Redundant query under a race. If two inserts ever observed stride == 0 at once, both would query and both would store the same value — harmless and self-correcting. Not worth a mutex.
Defensive default retained. Keep cmp.Or(detected, 1) so a pathological server value of 0 can't zero out the stride.
Session vs. global scope. The bare SELECT reads the session value, which each pooled connection inherits from global at connect time. This codebase never issues SET on the variable, so any pooled connection returns the representative value. Worth a one-line code comment.
Out of scope: innodb_autoinc_lock_mode. The assignIDs comment notes correctness also relies on lock mode = 2. Open question: validate it here too, or leave for a follow-up? Recommendation: leave out to keep scope tight.

Alternatives considered

Rejected Fallible auto-detecting constructor — NewMapper(ctx, handle, ...) (*Mapper, error) querying at construction. Cleanest-looking API, but it introduces a hard startup dependency on MySQL that doesn't exist today and would fail health checks during a transient DB outage. This is the approach your feedback correctly rules out.

Rejected Sentinel override — keep the stride param, auto-detect only when 0. Backward compatible but preserves the very argument we're trying to eliminate and muddles the contract.

Considered Detect during the recovery stage. Recovery already runs at startup and retries forever, but it only reads — coupling stride detection (a write-path concern) to it would be surprising, and recovery can complete without ever needing the stride. Lazy-on-write keeps the concern where it belongs.
